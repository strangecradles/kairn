**Status:** Planned  
**Depends on:** v1.7.0  
**Research:** ~/Projects/kairn-internal/RESEARCH-V3-BORIS.md

---

## The Problem

After `kairn describe` generates an environment, it shows:

```
  API keys needed (set these environment variables):

    export EXA_API_KEY="your-key-here"
      Get one at: https://dashboard.exa.ai/api-keys
```

But `export` doesn't persist across terminal sessions. The user has to manually create a `.env` file, add each key, and figure out how to get Claude Code to load it. Most users just skip this step and their MCP servers silently fail.

---

## v1.8.0 Solution — Interactive Key Collection + .env File

### The Flow

After writing the `.claude/` directory, if any selected tools need API keys, Kairn prompts the user immediately:

```
  ━━ API Keys ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Some tools need API keys to work. Enter them now,
  or press Enter to skip and add them later.

  EXA_API_KEY (Exa semantic search)
    Get one at: https://dashboard.exa.ai/api-keys
  › sk-exa-abc123...

  ✓ saved

  FIRECRAWL_API_KEY (Firecrawl web extraction)
    Get one at: https://firecrawl.dev/app/api-keys
  › (skipped)

  PERPLEXITY_API_KEY (Perplexity AI research)
    Get one at: https://perplexity.ai/settings/api
  › pplx-xyz789...

  ✓ saved

  ✓ Keys saved to .env (gitignored)
  ✓ .gitignore updated

  Skipped keys can be added later:
    Edit .env and fill in FIRECRAWL_API_KEY
```

### What Gets Written

**`.env` file (in project root):**
```bash
# Generated by Kairn — API keys for MCP servers
# Do NOT commit this file to git

EXA_API_KEY=sk-exa-abc123...
FIRECRAWL_API_KEY=
PERPLEXITY_API_KEY=pplx-xyz789...
```

**`.gitignore` (append if not already present):**
```
.env
```

**`.mcp.json` (already uses ${ENV_VAR} syntax — no change needed):**
```json
{
  "mcpServers": {
    "exa": {
      "env": {
        "EXA_API_KEY": "${EXA_API_KEY}"
      }
    }
  }
}
```

### How Claude Code Loads .env

Claude Code expands `${ENV_VAR}` from the shell environment. We need to ensure the `.env` is loaded when the user starts Claude Code. Two approaches:

**Approach A: SessionStart hook (recommended)**
Add a hook to the generated `settings.json` that loads `.env` on session start:

```json
{
  "hooks": {
    "SessionStart": [{
      "matcher": "startup",
      "hooks": [{
        "type": "command",
        "command": "if [ -f .env ]; then set -a; source .env; set +a; fi"
      }]
    }]
  }
}
```

Wait — SessionStart hooks can't inject env vars into Claude's process. The `CLAUDE_ENV_FILE` mechanism is needed.

**Approach B: CLAUDE_ENV_FILE (correct approach)**
SessionStart hooks can write to `$CLAUDE_ENV_FILE` to persist env vars:

```json
{
  "hooks": {
    "SessionStart": [{
      "matcher": "startup",
      "hooks": [{
        "type": "command",
        "command": "if [ -f .env ] && [ -n \"$CLAUDE_ENV_FILE\" ]; then grep -v '^#' .env | grep '=' >> \"$CLAUDE_ENV_FILE\"; fi"
      }]
    }]
  }
}
```

This reads `.env`, strips comments, and writes key=value pairs to Claude's env file so MCP servers can read them.

### Implementation

#### 1. Update `src/commands/describe.ts`

After writing the environment and showing env setup instructions, add an interactive key collection step:

```typescript
import { password } from "@inquirer/prompts";

// After writeEnvironment()...
if (summary.envSetup.length > 0 && !options.quick) {
  console.log(ui.section("API Keys"));
  console.log(ui.info("Some tools need API keys. Enter them now or press Enter to skip.\n"));

  const envEntries: string[] = [
    "# Generated by Kairn — API keys for MCP servers",
    "# Do NOT commit this file to git",
    "",
  ];

  let keysEntered = 0;
  const seen = new Set<string>();

  for (const env of summary.envSetup) {
    if (seen.has(env.envVar)) continue;
    seen.add(env.envVar);

    console.log(chalk.bold(`  ${env.envVar}`) + chalk.dim(` (${env.toolName})`));
    if (env.signupUrl) {
      console.log(chalk.dim(`    Get one at: ${env.signupUrl}`));
    }

    const value = await password({
      message: env.envVar,
      mask: "•",
    });

    if (value && value.trim()) {
      envEntries.push(`${env.envVar}=${value.trim()}`);
      console.log(chalk.green("  ✓ saved\n"));
      keysEntered++;
    } else {
      envEntries.push(`${env.envVar}=`);
      console.log(chalk.dim("  (skipped)\n"));
    }
  }

  // Write .env file
  const envPath = path.join(targetDir, ".env");
  await fs.writeFile(envPath, envEntries.join("\n") + "\n", "utf-8");

  // Update .gitignore
  const gitignorePath = path.join(targetDir, ".gitignore");
  let gitignore = "";
  try {
    gitignore = await fs.readFile(gitignorePath, "utf-8");
  } catch {}
  if (!gitignore.includes(".env")) {
    await fs.writeFile(
      gitignorePath,
      gitignore + (gitignore.endsWith("\n") ? "" : "\n") + ".env\n",
      "utf-8"
    );
  }

  console.log(chalk.green(`  ✓ ${keysEntered} key(s) saved to .env (gitignored)`));

  const skipped = summary.envSetup.filter(e => !seen.has(e.envVar)).length;
  if (skipped > 0) {
    console.log(chalk.dim(`  Skipped keys can be added later: edit .env`));
  }
}
```

## Future: Centralized Key Vault (~/.kairn/vault.json)

### The Vision (v2.x or v3.x)

Once a user enters an Exa API key for one project, they shouldn't have to enter it again for the next project. The key vault stores keys centrally:

```
~/.kairn/vault.json (encrypted)
{
  "EXA_API_KEY": "sk-exa-abc123...",
  "FIRECRAWL_API_KEY": "fc-xyz...",
  "PERPLEXITY_API_KEY": "pplx-789..."
}
```

**Flow with vault:**
```
kairn describe "research ML papers"

  ...environment generated...

  ━━ API Keys ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ✓ EXA_API_KEY — found in vault
  ✓ PERPLEXITY_API_KEY — found in vault
  ? BRAVE_API_KEY — not in vault
    Get one at: https://brave.com/search/api/
  › brv-abc123...
  ✓ saved to vault + .env

  ✓ All keys ready. 2 from vault, 1 new.
```

**The vault command:**
```bash
kairn vault list              # show stored keys (masked)
kairn vault add EXA_API_KEY   # add/update a key
kairn vault remove EXA_API_KEY
kairn vault export            # write all to .env in current dir
```

### Implementation Notes (for later)
- Encryption: AES-256-GCM using a key derived from machine ID
- File: `~/.kairn/vault.json`
- CLI: `kairn vault` subcommand
- Auto-populate .env from vault during `kairn describe` and `kairn optimize`
- Ties into v3.x MPP — vault becomes the fallback for non-MPP tools
