# v2.5.0 — Intent-Aware Harnesses (Two-Tier Evolving Router)

> "No commands to memorize, just describe what you want" — but project-specific, compiled from the codebase, and self-improving.

## Problem

Kairn generates powerful harnesses: CLAUDE.md, commands, rules, agents, skills. But users must **memorize command names** to use them. `/project:deploy`, `/project:test`, `/project:db-migrate` — if you don't know the exact command, you don't get the workflow.

OMC solves this with hardcoded regex keyword detection (14 categories, same for every project). But "deploy" means different things in a Next.js app vs. a Python ML pipeline vs. a Rust CLI. Generic keywords are the wrong abstraction.

## Solution

Kairn generates **intent-aware hooks** into every harness. Two tiers of routing, compiled from the project's actual workflows:

```
User types: "make sure everything works before I push"
                    │
                    ▼
          ┌─────────────────────┐
          │  TIER 1: Regex      │  <10ms, $0
          │  (command hook)     │
          │                     │
          │  "deploy" → deploy  │
          │  "test"   → test    │
          │  "lint"   → lint    │
          │  "push"   → ???     │  no match
          └─────────┬───────────┘
                    │ fallthrough
                    ▼
          ┌─────────────────────┐
          │  TIER 2: Haiku      │  ~2s, ~$0.001
          │  (prompt hook)      │
          │                     │
          │  Reads all available │
          │  commands + agents   │
          │  Classifies intent   │
          │  Returns workflow    │
          │                     │
          │  → /project:test    │
          │    then             │
          │    /project:lint    │
          │    then git push    │
          └─────────────────────┘
```

### Why Two Tiers?

| Property | Tier 1 (Regex) | Tier 2 (Haiku Prompt) |
|----------|---------------|----------------------|
| Latency | <10ms | 1-3s |
| Cost | $0 | ~$0.001/prompt |
| Accuracy | Exact match only | Semantic understanding |
| Works offline | Yes | No |
| Handles ambiguity | No | Yes |
| Evolves | Yes (proposer adds patterns) | Yes (prompt template tuned) |

Tier 1 catches the obvious. Tier 2 catches everything else. Over time, frequently-matched Tier 2 patterns get **promoted** to Tier 1 regexes — the harness learns the user's vocabulary.

### Generated Files

#### `.claude/settings.json` (hooks section)

```json
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "node \"$CLAUDE_PROJECT_DIR/.claude/hooks/intent-router.mjs\"",

          },

            "type": "prompt",

The user said something that didn't match any known command keyword.\n\nAvailable workflows:\n{{WORKFLOW_MANIFEST}}\n\nAvailable agents:\n{{AGENT_MANIFEST}}\n\nUser input: $PROMPT\n\nIf this maps to one or more workflows, return JSON:\n{\"additionalContext\": \"[INTENT ROUTED] Based on your request, use: /project:<command> — <description>\"}\n\nIf the user is asking a question or making a statement that doesn't need a workflow, return:\n{\"ok\": true}\n\nDo not activate workflows for questions like 'what does deploy do?' or 'how do I test?'.             "timeout": 15
          }
        ]
      }
    ]

```

The `{{WORKFLOW_MANIFEST}}` and `{{AGENT_MANIFEST}}` are **baked in at generation time** — the prompt template is compiled with the actual command/agent names and descriptions from the harness.

#### `.claude/hooks/intent-router.mjs` (Tier 1)

```javascript
#!/usr/bin/env node
/**
 * Kairn Intent Router — Tier 1 (Regex)
 * Generated by kairn describe. Project-specific keyword patterns.
 * 
 * If a keyword matches, injects a system-reminder pointing Claude
 * to the right command/workflow. If no match, falls through to
 * the Tier 2 prompt hook.
 */

import { readFileSync } from 'fs';

// Read prompt from stdin (Claude Code hooks API)
const input = readFileSync('/dev/stdin', 'utf-8');
let prompt = '';
try {
  const data = JSON.parse(input);
  prompt = data.prompt || '';
} catch { process.exit(0); }

// Sanitize: strip code blocks, URLs, file paths
const clean = prompt
  .replace(/```[\s\S]*?```/g, '')
  .replace(/`[^`]+`/g, '')
  .replace(/https?:\/\/\S+/g, '')
  .toLowerCase();

// Question filter: don't trigger on informational queries
const QUESTION_PATTERNS = [
  /\b(?:what(?:'s|\s+is)|how\s+(?:to|do\s+i)\s+use|explain|describe|tell\s+me\s+about)\b/i,
];
function isQuestion(text, position, length) {
  const start = Math.max(0, position - 80);
  const end = Math.min(text.length, position + length + 80);
  const ctx = text.slice(start, end);
  return QUESTION_PATTERNS.some(p => p.test(ctx));
}

// ═══════════════════════════════════════════════
// PROJECT-SPECIFIC KEYWORD PATTERNS
// Generated from: kairn describe analysis
// Last updated: {{GENERATION_TIMESTAMP}}
// ═══════════════════════════════════════════════

const PATTERNS = [
  // {{GENERATED_PATTERNS}}
  // Example for a Next.js + Vercel + Prisma project:
  // { pattern: /\b(deploy|ship|push\s+to\s+prod)/i,
  //   command: '/project:deploy',
  //   description: 'Deploy to Vercel production' },
  // { pattern: /\b(test|run\s+tests|check)/i,
  //   command: '/project:test',
  //   description: 'Run test suite (vitest)' },
  // { pattern: /\b(migrat|schema\s+change|db\s+update)/i,
  //   command: '/project:db-migrate',
  //   description: 'Run Prisma migration' },
];

// Match against patterns
for (const { pattern, command, description } of PATTERNS) {
  const match = clean.match(pattern);
  if (match && !isQuestion(clean, match.index, match[0].length)) {
    const output = {
      continue: true,
      hookSpecificOutput: {
        hookEventName: 'UserPromptSubmit',
        additionalContext: `[INTENT ROUTED] Based on your request, use: ${command} — ${description}\n\nUser's original request: ${prompt}\n\nExecute the command above to fulfill the user's intent.`
      }
    };
    console.log(JSON.stringify(output));
    process.exit(0);
  }
}

// No match — fall through to Tier 2 (prompt hook)
console.log(JSON.stringify({ continue: true, suppressOutput: true }));
```

### How Patterns Are Generated

During `kairn describe`, the compiler already analyzes:
- `package.json` scripts (npm run test, deploy, lint, etc.)
- Project structure (src/, tests/, migrations/, etc.)
- Framework detection (Next.js, Django, FastAPI, etc.)
- The commands it generates (deploy, test, lint, format, etc.)

For each generated command, the compiler produces a regex pattern:

```typescript
interface IntentPattern {
  pattern: string;          // regex source
  command: string;          // /project:command-name
  description: string;      // human-readable
  confidence: 'high' | 'medium';  // how specific the pattern is
  source: 'generated' | 'evolved'; // where it came from
}
```

**Generation heuristics:**
- Command name → direct keyword: `deploy` → `/\b(deploy)\b/`
- npm script names → keyword: `"test:e2e"` → `/\b(e2e|end.to.end)\b/`
- Framework-specific verbs: Next.js → `"build"`, Django → `"migrate"`, etc.
- Synonyms from a small static map: deploy → ship, push to prod; test → check, verify

The prompt template for Tier 2 is also compiled — it's not a generic "classify this intent" but a project-specific manifest:

```
Available workflows:
- /project:deploy — Deploy to Vercel production (vercel deploy --prod)
- /project:test — Run test suite (vitest run --reporter verbose)
- /project:lint — Run ESLint + Prettier check
- /project:db-migrate — Run Prisma schema migration
- /project:seed — Populate database with seed data

Available agents:
- @debugger — Root-cause analysis for build/runtime errors
- @reviewer — Code review with security focus
```

## Evolution Integration

### How `kairn evolve` Improves Intent Routing

The evolution loop already mutates harness files. Intent routing adds two new mutation targets:

1. **Tier 1 patterns** (in `intent-router.mjs`)
   - Add new regex patterns for commands that Tier 2 keeps routing
   - Tighten patterns that cause false positives
   - Remove patterns that never match

2. **Tier 2 prompt template** (in `settings.json`)
   - Refine the workflow manifest descriptions
   - Add examples of past successful routings
   - Adjust the classification threshold

### Self-Learning: The Pattern Promotion Flywheel

This is the transformative part. It runs **behind the scenes**, not as part of `kairn evolve run`:

```
Session 1: User says "ship it" → Tier 2 (Haiku) → /project:deploy
Session 2: User says "ship it" → Tier 2 (Haiku) → /project:deploy
Session 3: User says "ship it" → Tier 2 (Haiku) → /project:deploy
           ↓
           Pattern promotion: "ship it" → Tier 1 regex
           ↓
Session 4: User says "ship it" → Tier 1 (regex, 0ms, $0) → /project:deploy
```

#### Implementation: Background Learning Hook

A `PostToolUse` or `Stop` hook logs which Tier 2 routings were made:

```
.claude/hooks/intent-log.jsonl
{"timestamp":"...","prompt":"ship it","routed_to":"/project:deploy","tier":2}
{"timestamp":"...","prompt":"run the tests","routed_to":"/project:test","tier":2}
```

Periodically (on `SessionStart` or via a maintenance command), a learning step:

1. Reads `intent-log.jsonl`
2. Finds patterns that Tier 2 routed ≥3 times to the same command
3. Generates a regex that covers those prompts
4. Appends to Tier 1 PATTERNS array in `intent-router.mjs`
5. Clears the promoted entries from the log

This is the **data flywheel**: every Tier 2 call teaches Tier 1 what to catch next time.

#### Implementation: SessionStart Maintenance Hook

```json
{
  "SessionStart": [
    {
      "matcher": "*",
      "hooks": [
        {
          "type": "command",
          "command": "node \"$CLAUDE_PROJECT_DIR/.claude/hooks/intent-learner.mjs\"",
          "timeout": 10
        }
      ]
    }
  ]
}
```

The `intent-learner.mjs` script:
1. Reads `.claude/hooks/intent-log.jsonl`
2. Groups by `routed_to` command
3. For each command with ≥3 unique prompts → same route:
   - Extract common keywords across those prompts
   - Generate a regex pattern
   - Append to `PATTERNS` array in `intent-router.mjs`
   - Write a `.claude/hooks/intent-promotions.jsonl` audit log
4. Truncates processed entries from `intent-log.jsonl`

No LLM call needed for promotion — it's keyword extraction from a small set of known-good routings.

## Scope: What v2.5.0 Delivers

### Generation Pipeline (kairn describe)

1. **New compilation pass:** After generating commands/agents, generate intent patterns
   - Each command → 1-3 regex patterns (name, synonyms, framework verbs)
   - Compile Tier 2 prompt template with workflow + agent manifest

2. **New adapter output:** `writeEnvironment()` now also writes:
   - `.claude/hooks/intent-router.mjs` (Tier 1 engine + generated patterns)
   - `.claude/hooks/intent-learner.mjs` (background promotion logic)
   - `.claude/hooks/intent-log.jsonl` (empty, created on first Tier 2 hit)
   - Updated `.claude/settings.json` with `UserPromptSubmit` + `SessionStart` hooks

3. **New types:**
   ```typescript
   // In src/types.ts
   interface IntentPattern {
     pattern: string;           // regex source string
     command: string;           // /project:command-name
     description: string;       // human-readable
     source: 'generated' | 'evolved' | 'learned';
   }

   // Extend EnvironmentSpec.harness
   harness: {
     // ... existing fields ...
     hooks: Record<string, string>;          // hook script files
     intent_patterns: IntentPattern[];       // compiled patterns
     intent_prompt_template: string;         // Tier 2 prompt
   };
   ```

### Evolution Pipeline (kairn evolve)

4. **Proposer awareness:** The proposer reads intent patterns + Tier 2 prompt as part of the harness. It can propose mutations:
   - `add_intent_pattern` — add a new Tier 1 regex
   - `modify_intent_prompt` — refine the Tier 2 prompt template
   - Existing mutations still apply to commands/rules/agents

5. **Harness scope expansion:** `snapshotBaseline()` and `applyMutations()` now include:
   - `.claude/hooks/intent-router.mjs`
   - `.claude/settings.json` (hooks section)
   - `.claude/hooks/intent-learner.mjs`

6. **Intent-specific eval template:** New eval type in `templates.ts`:
   ```yaml
   - id: intent-routing
     template: intent-routing
     description: "User says 'deploy this' — does the harness route to /project:deploy?"
     setup: ""
     expected_outcome: "Claude executes /project:deploy without user specifying the command name"
     scoring: llm-judge
   ```

### Background Learning (self-improving, no CLI command)

7. **Tier 2 logging:** The Tier 2 prompt hook's `additionalContext` output is logged to `intent-log.jsonl` by a `PostToolUse` hook (or inline in the prompt hook output).

8. **Pattern promotion:** `intent-learner.mjs` runs on `SessionStart`, reads the log, promotes recurring Tier 2 → Tier 1 patterns.

9. **Audit trail:** `.claude/hooks/intent-promotions.jsonl` records every promotion with timestamp, original prompts, generated regex, and target command.

## What This Does NOT Do

- **Replace `kairn evolve`**: The background learning is lightweight (keyword extraction, no LLM). Full evolution (proposer + mutations + scoring) is still `kairn evolve run`.
- **Require a plugin**: Everything goes in `.claude/settings.json` (project scope). Kairn doesn't need to be a Claude Code plugin.
- **Work offline for Tier 2**: Tier 2 requires Anthropic API access. Tier 1 always works offline.
- **Add cost for Tier 1**: Regex matching is free. Tier 2 costs ~$0.001/prompt (~$0.10/day at 100 prompts).

## Cost Analysis

| Component | Per-prompt | Per-day (100 prompts) | Per-month |
|-----------|-----------|----------------------|-----------|
| Tier 1 (regex) | $0 | $0 | $0 |
| Tier 2 (Haiku) | ~$0.001 | ~$0.10 | ~$3 |
| Logging | $0 | $0 | $0 |
| Promotion (SessionStart) | $0 | $0 | $0 |

Total: ~$3/month maximum. Less as Tier 1 absorbs more patterns.
Compare: typical Claude Code usage is $20-100+/month.

**Users opt in automatically** — the hooks are generated by default. No config flag needed. The cost is negligible compared to normal Claude Code usage.

## Testing Strategy

1. **Unit tests:** Pattern generation from commands, sanitization, question filtering
2. **Integration tests:** Full hook flow — stdin JSON → pattern match → additionalContext output
3. **Tier 2 mock test:** Prompt template + mock Haiku response → correct routing
4. **Promotion test:** Log N entries → SessionStart → verify Tier 1 pattern added
5. **Regression test:** Ensure hooks don't break existing harness functionality
6. **E2E test:** Generate harness for test project → verify hooks in settings.json → simulate prompt

## Open Questions

1. **Hook chaining behavior:** If Tier 1 returns `additionalContext`, does Tier 2 still fire? Need to verify. If hooks chain (both fire), Tier 1 needs to signal "handled" so Tier 2 skips. If they short-circuit on first `additionalContext`, this works automatically.

2. **Intent-learner frequency:** On every `SessionStart`? Or only after N new log entries? SessionStart seems right — it's cheap (no LLM), and users start sessions frequently enough.

3. **Pattern quality:** Auto-generated regexes from keyword extraction may be too broad or too narrow. The evolution loop should catch this, but the initial generation heuristics matter for first impressions.

4. **Multi-command routing:** "make sure everything works before pushing" → test + lint + push. Tier 2 can return multiple commands. How does `additionalContext` express a sequence? Probably: "Use /project:test, then /project:lint, then git push" as natural language Claude interprets.

5. **Prompt hook access to user input:** The docs show `$PROMPT` in prompt hooks and `data.prompt` in command hooks. Need to verify the exact variable name for UserPromptSubmit prompt hooks.
