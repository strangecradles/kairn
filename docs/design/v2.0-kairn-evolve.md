# v2.0 Design Doc — Kairn Evolve (Automated Harness Optimization)

> Kairn generates agent environments. Kairn Evolve makes them better — automatically. Run your agent on real tasks, log everything, diagnose failures, mutate the harness, and repeat until performance plateaus.

**Status:** Planned  
**Depends on:** v1.14.0  
**Primary inspiration:** [Meta-Harness](https://yoonholee.com/meta-harness/) (Lee et al., Stanford IRIS Lab, 2026)  
**Secondary references:** [DSPy](https://github.com/stanfordnlp/dspy), [OpenEvolve](https://github.com/algorithmicsuperintelligence/openevolve), [TextGrad](https://github.com/zou-group/textgrad), [OPRO](https://github.com/google-deepmind/opro)

---

## The Problem

Kairn v1 generates environments from a natural language description. The quality of the output depends entirely on the compilation prompt and model — it's a one-shot process. If the generated CLAUDE.md has a bad instruction, a missing tool, or a poorly-structured command, the user discovers this only after working with the environment and getting frustrated.

There's no feedback loop. The environment never improves based on actual usage.

Meta-Harness (Stanford, March 2026) proved that an outer-loop optimizer can dramatically improve agent scaffolds. Their key insight: provide the optimizer with **full execution traces** (not just scalar scores) so it can perform **causal diagnosis** of failures. Their system improved a Haiku 4.5 agent to #1 on TerminalBench-2 in just 7 iterations.

The landscape of prompt/harness optimizers:

| System | What it optimizes | Feedback type | Context/iter | Search strategy |
|--------|------------------|---------------|-------------|-----------------|
| OPRO | Prompt text | Score pairs | ~2K tok | History window |
| TextGrad | Any text variable | LLM "gradients" | ~15K tok | Gradient descent |
| DSPy | Prompt modules | Task metrics | ~50K tok | Bootstrap + MIPROv2 |
| OpenEvolve | Full code files | Execution results | ~22K tok | Evolutionary (MAP-Elites) |
| **Meta-Harness** | **Agent scaffold** | **Full traces** | **~10M tok** | **Agentic causal reasoning** |
| **Kairn Evolve** | **.claude/ directory** | **Full traces** | **~10M tok** | **Agentic (Meta-Harness-inspired)** |

Kairn Evolve adapts the Meta-Harness approach to the specific domain of Claude Code agent environments (`.claude/` directories).

#### 2.0.0-B: Eval Template Menu & Auto-Generation

Evals are **auto-generated** by default. `kairn evolve init` reads the project's `.claude/CLAUDE.md` and project structure, selects relevant eval templates, and uses the LLM to instantiate concrete, project-specific tasks.

**Why auto-generate?** The environment was generated by Kairn, so Kairn already knows the intent, tech stack, workflow type, and tools. Hand-writing evals creates friction and most users won't do it. But the evals must be project-specific — generic benchmarks (SWE-bench, TerminalBench) test model capability, not harness quality.

**Two-layer design:**

**Layer 1 — Eval Templates** (shipped with Kairn, the "menu"):

| Template | Tests | Example instantiation |
|----------|-------|-----------------------|
| `add-feature` | Can the agent add a new capability? | "Add a /health endpoint returning { status: 'ok' }" |
| `fix-bug` | Can the agent diagnose and fix? | "Fix pagination returning duplicate page 1 results" |
| `refactor` | Can the agent restructure code? | "Extract DB queries into a data access layer" |
| `test-writing` | Can the agent write tests? | "Add integration tests for the auth middleware" |
| `config-change` | Can the agent update config/infra? | "Add ESLint with the project's style rules" |
| `documentation` | Can the agent write docs? | "Generate API docs for all /api/ endpoints" |

**Layer 2 — LLM Instantiation** (project-specific):

The LLM reads `.claude/CLAUDE.md`, scans the project structure (tech stack, existing files, scripts), and generates 3-5 concrete tasks by selecting the most relevant templates and filling in project-specific details.

```
kairn evolve init flow:
  1. Read .claude/CLAUDE.md → extract intent, tech stack, workflow type
  2. Scan project structure → languages, frameworks, test setup, scripts
  3. Select 3-5 eval templates from menu based on workflow type
  4. LLM generates concrete tasks from templates + project context
  5. Write tasks.yaml → user can review/edit before proceeding
  6. Interactive: "Add another eval? [y/n]" → pick template → LLM fills details
```

#### 2.0.0-C: Task Runner

The runner executes a single task against the current `.claude/` environment:

```typescript
// src/evolve/runner.ts

interface TaskRun {
  taskId: string;
  iteration: number;
  traceDir: string;  // .kairn-evolve/traces/{iteration}/{taskId}/
}

async function runTask(task: Task, harness: HarnessSnapshot, run: TaskRun): Promise<TaskResult> {
  // 1. Create isolated workspace (git worktree or temp directory)
  const workspace = await createIsolatedWorkspace(task);

  // 2. Copy the harness (.claude/) into the workspace
  await copyHarness(harness, workspace);

  // 3. Run task.setup commands
  await executeSetup(task.setup, workspace);

  // 4. Spawn Claude Code with the task description
  //    Capture ALL output: stdout, stderr, tool calls, file changes
  const trace = await spawnClaudeCode({
    workspace,
    instruction: task.description,
    timeout: task.timeout,
    captureTrace: true,
  });

  // 5. Write trace to filesystem
  await writeTrace(run.traceDir, trace);

  // 6. Score the result
  const score = await scoreTask(task, workspace, trace);
  await writeJson(`${run.traceDir}/score.json`, score);

  // 7. Clean up workspace
  await cleanupWorkspace(workspace);

  return { taskId: task.id, score, traceDir: run.traceDir };
}
```

### v2.1.0 — The Evolution Loop

This is the core innovation — the outer loop that makes environments self-improving.

#### 2.1.0-A: Proposer Agent

The proposer is an LLM (Opus) that reads the full trace filesystem and proposes harness changes:

```typescript
// src/evolve/proposer.ts

async function propose(
  iteration: number,
  traceDir: string,        // .kairn-evolve/traces/{iteration}/
  currentHarness: string,  // .kairn-evolve/iterations/{iteration}/harness/
  history: IterationHistory[],
): Promise<Proposal> {
  // Build the proposer's context:
  // 1. Current harness files (CLAUDE.md, commands/, rules/, agents/)
  // 2. All traces from this iteration (stdout, tool_calls, scores)
  // 3. History: previous proposals, their diffs, and score changes
  // 4. The task definitions (what we're optimizing for)

  const prompt = buildProposerPrompt({
    currentHarness,
    traces: await loadTraces(traceDir),
    history,
    taskDefs: await loadTasks(),
  });

  // Call the proposer model
  const response = await callLLM({
    model: config.proposerModel,
    system: PROPOSER_SYSTEM_PROMPT,
    user: prompt,
    maxTokens: 8192,
  });

  // Parse the response into structured mutations
  return parseProposal(response);
}
```

The proposer system prompt (key piece — this is what makes it Meta-Harness-like):

```markdown
You are an expert agent environment optimizer. Your job is to improve a Claude Code
agent environment (.claude/ directory) based on execution traces from real tasks.

## What You Have Access To

1. **Current harness:** The .claude/ directory files (CLAUDE.md, commands/, rules/, agents/)
2. **Execution traces:** Full stdout/stderr, tool call sequences, file changes, and scores
   for every task run against this harness
3. **History:** Previous iterations' proposals, diffs, and resulting score changes

## Your Task

Analyze the traces to identify WHY tasks fail or underperform. Then propose specific,
minimal changes to the harness files that will fix those failures.

## Diagnosis Process

1. For each failed/low-scoring task:
   a. Read the full trace (stdout, tool calls, file changes)
   b. Identify the ROOT CAUSE: bad instruction in CLAUDE.md? Missing tool? Wrong rule?
      Poorly structured command? Missing context?
   c. Trace the failure back to a specific harness decision
   d. Propose a fix

2. For each successful task:
   a. Note what worked well
   b. Ensure proposed changes don't break what's working

3. Check history for counterfactual evidence:
   - Did a previous change help some tasks but hurt others?
   - What's the common thread in regressions?

## Output Format

Return a JSON object with:
{
  "reasoning": "Your full causal analysis...",
  "mutations": [
    {
      "file": "CLAUDE.md",
      "action": "replace",
      "old_text": "...",
      "new_text": "...",
      "rationale": "Why this change fixes task X"
    },
    {
      "file": "commands/develop.md",
      "action": "replace",
      "old_text": "...",
      "new_text": "...",
      "rationale": "..."
    }
  ],
  "expected_impact": {
    "task-auth": "+15% — fixes the missing middleware instruction",
    "task-pagination": "neutral — unrelated",
    "task-refactor": "+5% — better separation guidance"
  }
}

## Rules

- MINIMAL changes only. Don't rewrite the entire CLAUDE.md.
- Each mutation must have a clear rationale tied to a specific trace observation.
- Never remove something that's working for another task.
- If a previous iteration's change caused a regression, REVERT it.
- Prefer ADDITIVE changes over replacements when possible.
```

#### 2.1.0-B: Harness Diff Engine

Apply the proposed mutations to the harness:

```typescript
// src/evolve/mutator.ts

async function applyMutations(
  currentHarness: string,
  mutations: Mutation[],
): Promise<{ newHarness: string; diffPatch: string }> {
  // 1. Copy current harness to new iteration directory
  const newHarness = await copyHarnessDir(currentHarness, nextIteration);

  // 2. Apply each mutation
  for (const mutation of mutations) {
    if (mutation.action === 'replace') {
      await patch(
        path.join(newHarness, mutation.file),
        mutation.old_text,
        mutation.new_text,
      );
    } else if (mutation.action === 'add_section') {
      await appendToFile(
        path.join(newHarness, mutation.file),
        mutation.new_text,
      );
    } else if (mutation.action === 'create_file') {
      await writeFile(
        path.join(newHarness, mutation.file),
        mutation.new_text,
      );
    }
  }

  // 3. Generate unified diff for the iteration log
  const diffPatch = await generateDiff(currentHarness, newHarness);

  return { newHarness, diffPatch };
}
```

#### 2.1.0-C: The Main Loop

```typescript
// src/evolve/loop.ts

async function evolve(config: EvolveConfig): Promise<EvolveResult> {
  const history: IterationHistory[] = [];

  for (let iter = 0; iter < config.maxIterations; iter++) {
    console.log(`\n━━ Iteration ${iter} ━━━━━━━━━━━━━━━━━━━━━━━━━━━━`);

    // 1. EVALUATE: Run all tasks against current harness
    const results = await evaluateAll(iter);
    const aggregateScore = aggregate(results);
    console.log(`  Score: ${aggregateScore.toFixed(1)}%`);

    // 2. CHECK: Did we regress? Rollback if so.
    if (iter > 0 && aggregateScore < history[iter - 1].score) {
      console.log(`  ⚠ Regression detected. Rolling back.`);
      await rollback(iter);
      continue;
    }

    // 3. CHECK: Are we at 100%? Stop if so.
    if (aggregateScore >= 100) {
      console.log(`  ✔ Perfect score. Stopping.`);
      break;
    }

    // 4. DIAGNOSE + MUTATE: Proposer reads traces, suggests changes
    const proposal = await propose(iter, results, history);
    console.log(`  Proposed ${proposal.mutations.length} mutations`);

    // 5. APPLY: Mutate the harness
    const { newHarness, diffPatch } = await applyMutations(
      getCurrentHarness(iter),
      proposal.mutations,
    );

    // 6. LOG: Save everything
    history.push({
      iteration: iter,
      score: aggregateScore,
      proposal,
      diffPatch,
      results,
    });
    await saveIterationLog(iter, history[iter]);

    // 7. ADVANCE: Set the new harness as current
    await setCurrentHarness(newHarness);
  }

  return { history, bestIteration: findBest(history) };
}
```

### v2.2.1 — Proposer JSON Fix + Mutation Scope Expansion (Bugfix Release)

Discovered during Kairn-on-Kairn test runs (2026-04-01). Two rounds of testing revealed:

**Test Run 1** (pre-bugfix): 3 iterations, 6 tasks. Baseline 0.3% → Best 12.0%. Proposer failed silently on all iterations. All score changes were random variance.

**Test Run 2** (post-bugfix for runner/scorer/progress): 3 iterations, 4 tasks. Baseline 93.0% → Best 96.3%. 3/4 tasks pass at 100%. But proposer STILL fails — returns English analysis instead of JSON:
```
Error: "Proposer returned invalid JSON: Looking at the traces, I need to analyze why each task scored as it did..."
```
No mutations were ever applied across all iterations. The score improvement (93→96.3) is random Claude Code nondeterminism on the `fix-and-commit-workflow` task.

**Root causes:**
1. **#1 BLOCKER — Proposer JSON failure:** `callLLM()` doesn't use JSON mode or assistant prefill. The system prompt says "Return ONLY valid JSON" but the LLM writes English prose first. The parser only handles code fences, not JSON embedded in text.
2. Mutations are additive-only (no way to remove bloat)
3. `.mcp.json` is outside harness scope (MCP servers can't be evolved)
4. Proposer prompt biases toward additions even when removals are needed

This is a patch release — no new CLI commands, just internal fixes to make the loop actually evolve.

#### Proposer JSON Fix (the #1 blocker)

Two layers of defense:

**Layer 1: Assistant prefill in callLLM()**

Add `jsonMode` option to `callLLM()`. For Anthropic, use the assistant prefill technique:

```typescript
// src/llm.ts — when jsonMode is true
if (jsonMode && config.provider === "anthropic") {
  // Append JSON instruction to system prompt
  const jsonSystem = systemPrompt + "\n\nIMPORTANT: Your response must be ONLY a valid JSON object. No text before or after. Start with { and end with }.";

  const response = await client.messages.create({
    model: config.model,
    max_tokens: maxTokens,
    system: jsonSystem,
    messages: [
      { role: "user", content: userMessage },
      { role: "assistant", content: "{" }  // Forces JSON start
    ],
  });
  // Prepend the { that was used as prefill
  return "{" + textBlock.text;
}

// For OpenAI-compatible providers:
response_format: { type: "json_object" }
```

**Layer 2: Robust JSON extraction in parser**

Make `parseProposerResponse()` tolerant of prose-wrapped JSON:

```typescript
// src/evolve/proposer.ts — enhanced parseProposerResponse
function extractJsonFromProse(raw: string): string {
  // Strategy 1: Direct parse
  try { JSON.parse(raw); return raw; } catch {}

  // Strategy 2: Strip code fences
  const fenced = raw.match(/```(?:json)?\s*\n?([\s\S]*?)\n?\s*```/);
  if (fenced) { try { JSON.parse(fenced[1]); return fenced[1]; } catch {} }

  // Strategy 3: Find first { to last } (handles prose before/after)
  const firstBrace = raw.indexOf('{');
  const lastBrace = raw.lastIndexOf('}');
  if (firstBrace !== -1 && lastBrace > firstBrace) {
    const extracted = raw.slice(firstBrace, lastBrace + 1);
    try { JSON.parse(extracted); return extracted; } catch {}
  }

  throw new Error(`No valid JSON found in response: ${raw.slice(0, 200)}`);
}
```

#### Bug 5 Fix: Balanced Proposer Prompt

Current proposer system prompt says "Prefer ADDITIVE changes over replacements when possible." This biases the loop toward growth-only optimization, which causes harness bloat over iterations.

Replace the rules section:

```markdown
## Rules
- MINIMAL changes only. Don't rewrite the entire CLAUDE.md.
- Each mutation must have a clear rationale tied to a specific trace observation.
- Never remove something that's working for another task.
- If a previous iteration's change caused a regression, REVERT it.
- Consider BOTH additions AND removals:
  - If traces show the agent ignoring harness sections → remove or simplify them.
  - If traces show the agent missing instructions → add them.
  - If MCP servers are unused → consider removing them from .mcp.json.
  - Leaner harnesses perform better (fewer tokens consumed = more context for the task).
- Available mutation actions: replace, add_section, create_file, delete_section, delete_file
```

## Connection to Research

### What we borrow from Meta-Harness
- Full trace filesystem (not compressed feedback)
- Agentic proposer with causal reasoning
- Additive-first mutation strategy
- Rollback on regression

### What we borrow from DSPy
- Task definition with metrics (their "signatures + metrics")
- Modular scoring (their "teleprompters")
- Train/test split for generalization

### What we borrow from OpenEvolve
- Population-based search as an advanced option
- Quality-diversity optimization (don't just find one good harness — explore the space)
- Cost tracking and budget management

### What we borrow from TextGrad
- The idea of "textual gradients" — but we implement this as the proposer's causal reasoning
- Per-component feedback (which specific file/section caused the failure?)

### What's unique to Kairn Evolve
- Operates on `.claude/` directories (not arbitrary code)
- Mutations are constrained to the Claude Code harness format
- Integrates with an existing generation pipeline (Kairn v1)
- User-facing CLI with familiar `kairn` ergonomics
- Designed for Claude Code specifically (not model-agnostic)
