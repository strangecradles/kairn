# v2.7.0 — Structured Harness IR

> Raw Markdown mutation will corrupt formatting, accumulate contradictions, and break as files grow. A structured intermediate representation makes mutations composable, diffing meaningful, and format migration tractable.

## Problem

Today, the entire Kairn pipeline operates on raw strings:

1. **Compilation** (Pass 2): The LLM generates `HarnessContent` with `claude_md: string` and `commands: Record<string, string>` — raw Markdown blobs.
2. **Mutation** (`mutator.ts`): The evolve proposer emits text operations (`replace`, `add_section`, `delete_section`) that do string surgery on files. `replace` fails silently if the exact substring isn't found. `add_section` blindly appends. There's no validation that the result is a coherent harness.
3. **Diffing** (`generateDiff`): Compares files character-by-character. A section reorder looks like a complete rewrite. Moving a rule from one file to another looks like a deletion + creation with no connection between them.
4. **Complexity measurement** (`regularization.ts`): Counts lines and files. Can't distinguish a 10-line section that adds real value from a 10-line section that duplicates an existing rule.

These problems compound in the evolution loop. After 5 iterations of text mutations, CLAUDE.md can have duplicate sections, contradictory instructions, and broken formatting — even when each individual mutation was reasonable. The proposer wastes tokens diagnosing formatting artifacts rather than real performance issues.

**Strategically**, raw string manipulation locks Kairn to `.claude/` output. The same harness knowledge (rules, commands, agents) could render to Cursor rules, Codex configs, or any future runtime — but only if that knowledge is stored structurally, not as pre-formatted Markdown.

## Solution

Introduce a **Harness IR** (Intermediate Representation) — a typed data model that sits between intent and output files:

```
Intent → Compiler → HarnessIR → Renderer → .claude/ files
                                          → .cursor/ files  (future)
                                          → .codex/ files   (future)

Existing .claude/ → Parser → HarnessIR → Mutations → HarnessIR → Renderer → .claude/
```

The IR is the single source of truth. Mutations operate on the IR tree. Renderers produce output files deterministically from the IR. Parsers ingest existing harnesses back into the IR.

## IR Type Model

```typescript
/** Top-level harness intermediate representation */
interface HarnessIR {
  meta: HarnessMeta;
  sections: Section[];          // CLAUDE.md content sections
  commands: CommandNode[];      // .claude/commands/*.md
  rules: RuleNode[];            // .claude/rules/*.md
  agents: AgentNode[];          // .claude/agents/*.md
  skills: SkillNode[];          // .claude/skills/*.md
  docs: DocNode[];              // .claude/docs/*.md
  hooks: HookNode[];            // .claude/hooks/*.mjs
  settings: SettingsIR;         // .claude/settings.json
  mcpServers: McpServerNode[];  // .mcp.json entries
  intents: IntentNode[];        // intent routing patterns
}

/** Project metadata extracted from the harness */
interface HarnessMeta {
  name: string;
  purpose: string;
  techStack: {
    language: string;
    framework?: string;
    buildTool?: string;
    testRunner?: string;
    packageManager?: string;
  };
  autonomyLevel: 1 | 2 | 3 | 4;
}

/** A semantic section within CLAUDE.md */
interface Section {
  id: string;           // stable identifier: "purpose", "tech-stack", "commands", etc.
  heading: string;      // rendered heading: "## Purpose"
  content: string;      // markdown body (no heading prefix)
  order: number;        // rendering order within CLAUDE.md
}

/** A slash command */
interface CommandNode {
  name: string;         // "dev", "test", "deploy"
  description: string;  // one-line summary
  content: string;      // full markdown body
}

/** A behavioral rule */
interface RuleNode {
  name: string;         // "security", "typescript", "testing"
  paths?: string[];     // path scoping from YAML frontmatter
  content: string;      // rule body
}

/** An agent definition */
interface AgentNode {
  name: string;         // "architect", "implementer"
  model?: string;       // "opus", "sonnet", "haiku"
  disallowedTools?: string[];
  content: string;      // agent instructions
}

/** A skill definition */
interface SkillNode {
  name: string;
  content: string;
}

/** A documentation file */
interface DocNode {
  name: string;
  content: string;
}

/** A hook script */
interface HookNode {
  name: string;         // "intent-router", "intent-learner"
  content: string;      // JavaScript source
  type: 'command' | 'prompt';
}

/** Settings.json structured as IR */
interface SettingsIR {
  statusLine?: { command: string };
  hooks: {
    PreToolUse?: HookEntry[];
    PostToolUse?: HookEntry[];
    UserPromptSubmit?: HookEntry[];
    SessionStart?: HookEntry[];
  };
  denyPatterns?: string[];
  raw: Record<string, unknown>;  // pass-through for unstructured fields
}

interface HookEntry {
  matcher: string;
  hooks: Array<{
    type: 'command' | 'prompt';
    command?: string;
    prompt?: string;
    timeout?: number;
  }>;
}

/** An MCP server configuration */
interface McpServerNode {
  id: string;           // server name
  command: string;
  args: string[];
  env?: Record<string, string>;
}

/** An intent routing pattern */
interface IntentNode {
  commandName: string;
  patterns: string[];   // regex patterns
  priority: number;
}
```

### Why This Shape

**Sections, not a single string.** CLAUDE.md is currently a single `string` field. The IR breaks it into semantic sections with stable IDs. This means:
- A mutation can target "update the conventions section" without finding/replacing raw text
- Two mutations to different sections never conflict
- Section reordering is a single field change, not a massive text diff
- Renderers can enforce section ordering rules per runtime

**Nodes with metadata, not `Record<string, string>`.** Commands currently live in `Record<string, string>` — name → raw content. The IR adds structured metadata (description, model, paths) that enables:
- Semantic diffing ("agent X changed its model from Sonnet to Opus")
- Validation ("rule Y claims to scope to `src/api/**` but that path doesn't exist")
- Smart mutation ("add a disallowed tool to this agent" vs. text surgery on the markdown)

**Settings as structured data.** `settings.json` is currently `Record<string, unknown>` — opaque JSON. The IR makes hooks, deny patterns, and status line first-class fields. This lets the evolution loop reason about settings structurally ("this harness has 4 hooks but none of them run tests").

## Parser: .claude/ → HarnessIR

The parser reads an existing `.claude/` directory and produces a `HarnessIR`:

```
parseHarness(harnessPath: string): Promise<HarnessIR>
```

**CLAUDE.md parsing strategy:**
1. Split on `^## ` heading boundaries
2. Assign section IDs by matching heading text to known patterns:
   - "Purpose" / "About" → `purpose`
   - "Tech Stack" / "Technology" → `tech-stack`
   - "Commands" / "Key Commands" → `commands`
   - "Architecture" → `architecture`
   - "Conventions" → `conventions`
   - "Verification" → `verification`
   - "Known Gotchas" → `gotchas`
   - Unrecognized → `custom-{slug}`
3. Extract `HarnessMeta.techStack` from the tech stack section (language, framework, etc.)
4. Preserve original heading text and order

**Rules/agents parsing:** Read YAML frontmatter (if present) for `paths:`, `model:`, `disallowedTools:`, then treat the rest as content.

**Settings parsing:** Read `settings.json`, extract known structures (hooks, statusLine, deny patterns), preserve the rest in `raw`.

**MCP parsing:** Read `.mcp.json`, decompose `mcpServers` into individual `McpServerNode` entries.

### Round-trip fidelity

The parser must satisfy: `render(parse(dir)) ≈ dir`. Not byte-identical (whitespace normalization is fine), but semantically identical. This is tested by a round-trip integration test.

## Renderer: HarnessIR → Files

```
renderHarness(ir: HarnessIR): Map<string, string>
renderHarnessToDir(ir: HarnessIR, targetDir: string): Promise<string[]>
```

The renderer is deterministic — the same IR always produces the same files. This is the **only** place that knows about output format.

**CLAUDE.md rendering:**
1. Sort sections by `order`
2. For each section: `{heading}\n\n{content}`
3. Join with `\n\n---\n\n` or `\n\n`

**Settings rendering:** Merge structured fields (hooks, statusLine, denyPatterns) with `raw` pass-through. Output as formatted JSON.

**Commands/rules/agents/skills/docs:** `{name}.md` in the appropriate directory. Agents with `model:` frontmatter get YAML headers prepended.

### Future renderers

The renderer interface is abstract enough that adding new runtimes means writing a new renderer function, not changing the IR:

```typescript
type HarnessRenderer = (ir: HarnessIR) => Map<string, string>;

const renderers: Record<string, HarnessRenderer> = {
  'claude-code': renderClaudeCode,
  'cursor': renderCursor,      // future
  'codex': renderCodex,        // future
};
```

## IR Mutations

Replace the current text-based `Mutation` type with typed IR mutations:

```typescript
type IRMutation =
  // Section mutations (CLAUDE.md)
  | { type: 'update_section'; sectionId: string; content: string; rationale: string }
  | { type: 'add_section'; section: Section; rationale: string }
  | { type: 'remove_section'; sectionId: string; rationale: string }
  | { type: 'reorder_section'; sectionId: string; newOrder: number; rationale: string }
  // Command mutations
  | { type: 'add_command'; command: CommandNode; rationale: string }
  | { type: 'update_command'; name: string; content: string; rationale: string }
  | { type: 'remove_command'; name: string; rationale: string }
  // Rule mutations
  | { type: 'add_rule'; rule: RuleNode; rationale: string }
  | { type: 'update_rule'; name: string; content: string; rationale: string }
  | { type: 'remove_rule'; name: string; rationale: string }
  // Agent mutations
  | { type: 'add_agent'; agent: AgentNode; rationale: string }
  | { type: 'update_agent'; name: string; changes: Partial<AgentNode>; rationale: string }
  | { type: 'remove_agent'; name: string; rationale: string }
  // MCP mutations
  | { type: 'add_mcp_server'; server: McpServerNode; rationale: string }
  | { type: 'remove_mcp_server'; id: string; rationale: string }
  // Settings mutations
  | { type: 'update_settings'; path: string; value: unknown; rationale: string }
  // Legacy text fallback (for .mcp.json or unstructured files)
  | { type: 'raw_text'; file: string; action: 'replace' | 'add_section' | 'create_file' | 'delete_section' | 'delete_file'; oldText?: string; newText: string; rationale: string };
```

**Applying mutations:** Each mutation operates directly on the IR tree:
- `update_section`: Find section by ID, replace content
- `add_command`: Append to `ir.commands`, check for name collisions
- `remove_rule`: Filter out from `ir.rules` by name
- No string matching, no fragile find-and-replace

**Legacy compatibility:** The `raw_text` mutation type preserves backward compatibility. Existing proposer prompts continue to work. The proposer is gradually migrated to emit IR mutations in later versions.

## IR Diff Engine

```
diffIR(before: HarnessIR, after: HarnessIR): IRDiff
```

Produces a structured diff that's meaningful at the harness level:

```typescript
interface IRDiff {
  sections: {
    added: Section[];
    removed: Section[];
    modified: Array<{ id: string; before: string; after: string }>;
    reordered: Array<{ id: string; oldOrder: number; newOrder: number }>;
  };
  commands: {
    added: CommandNode[];
    removed: string[];
    modified: Array<{ name: string; before: string; after: string }>;
  };
  rules: { added: RuleNode[]; removed: string[]; modified: Array<{ name: string; before: string; after: string }> };
  agents: { added: AgentNode[]; removed: string[]; modified: Array<{ name: string; changes: string }> };
  mcpServers: { added: McpServerNode[]; removed: string[] };
  settings: { changes: Array<{ path: string; before: unknown; after: unknown }> };
}
```

This replaces the current character-level `generateDiff()` in `mutator.ts`. The proposer gets semantically meaningful diffs instead of raw text patches.

## Integration Points

### Evolution Loop (`loop.ts`)

1. At loop start: `parse` baseline harness → `HarnessIR`
2. Proposer receives the IR (serialized) instead of raw file contents
3. Proposer returns `IRMutation[]` instead of text `Mutation[]`
4. `applyIRMutations(ir, mutations)` produces new IR
5. `renderHarness(newIR)` writes iteration harness to disk
6. `diffIR(oldIR, newIR)` produces the iteration diff log
7. KL regularization operates on IR node counts (sections, commands, rules) — more precise than line counting

### Compiler (`compile.ts`)

1. Pass 2 LLM output is parsed into IR (not raw strings)
2. Pass 3 builds settings/MCP from IR (already partially structured)
3. `renderHarness(ir)` replaces `writeEnvironment(spec)` for file output
4. `EnvironmentSpec.harness` gains an `ir: HarnessIR` field alongside the raw strings (migration period)

### Proposer (`proposer.ts`)

**Phase 1 (this version):** Proposer still reads raw files and returns text mutations. A translation layer converts text mutations into the closest IR mutation. The raw_text fallback handles anything that doesn't map cleanly.

**Phase 2 (future):** Proposer system prompt is updated to emit IR mutations directly. The proposer reads the IR JSON, not raw file contents. This eliminates the translation layer and the raw_text fallback.

## Migration Path

Existing `.claude/` directories (hand-written or generated by older Kairn versions) must parse into the IR correctly. The migration is:

```
Existing .claude/ → parseHarness() → HarnessIR → renderHarness() → upgraded .claude/
```

The round-trip should be semantically lossless:
- All content preserved
- Sections may be normalized (consistent heading format)
- Whitespace may change
- YAML frontmatter parsed and re-serialized consistently

`kairn optimize` can offer a `--migrate` flag to upgrade existing harnesses to IR-rendered format.

## File Structure

```

  parser.ts         — parseHarness(): .claude/ → HarnessIR

    translate.test.ts
```

## Non-Goals (This Version)

- **Multi-runtime renderers.** Only Claude Code renderer ships in v2.7.0. Cursor/Codex renderers are future work — but the IR is designed to support them.
- **IR-native proposer.** The proposer still works with text in v2.7.0. A translation layer bridges text mutations → IR mutations. Full IR-native proposer is v2.8.0+.
- **Schema validation.** The IR is structurally typed but not schema-validated against Claude Code's runtime spec. That's a future quality gate.
- **CLAUDE.md section merging.** Two IR mutations to the same section both update the full content — they don't try to merge at the paragraph level. That's a text merge problem we're explicitly avoiding.

## Success Criteria

- [ ] `parseHarness()` correctly parses real `.claude/` directories (hand-written and Kairn-generated)
- [ ] `renderHarness()` produces identical-ish output from parsed IR (round-trip test)
- [ ] Evolution loop works with IR: parse → mutate → render → evaluate cycle
- [ ] IR mutations are type-safe: no mutation can produce invalid tree structure
- [ ] `diffIR()` produces human-readable semantic diffs
- [ ] Existing text-based mutations still work via translation layer
- [ ] KL regularization uses IR node counts instead of line counts
- [ ] `npm run build`, `npm test`, `npx tsc --noEmit` all pass
