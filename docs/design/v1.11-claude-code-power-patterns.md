# v1.11.0 Design Doc — Claude Code Power Patterns & Orchestration

> Adapt the real Claude Code internal prompts and engineering patterns into Kairn's generated environments, culminating in `/project:develop` as a multi-phase orchestrator. Fix internal documentation bloat by making docs actionable.

**Status:** Planned  
**Depends on:** v1.10.1  
**Source repo:** https://github.com/Piebald-AI/claude-code-system-prompts

---

## The Core Idea

Claude Code v2.1.88 contains 110+ internal prompt strings that are conditionally injected based on context. Many of these implement powerful patterns that are NOT exposed as public slash commands — they're orchestrated internally via subagent spawning, tool gating, and prompt injection.

Kairn can **adapt** these patterns into generated slash commands, agents, and CLAUDE.md directives. We're not copying — we're translating internal orchestration logic into the harness format that Claude Code's public API supports.

This version has three layers:
1. **Orchestration Layer** — the new `/project:develop` command
2. **Building Blocks** — new commands and agents adapted from internal Claude Code features
3. **Engineering Principles** — behavioral directives embedded into every generated CLAUDE.md

---

## Layer 1: Orchestration Layer

### /project:develop — Full Development Pipeline

This command replaces the monolithic `/project:ship` with a phase-based orchestration of specialized subagents, ensuring high-quality output and active documentation. User interaction (e.g., confirmations, spec refinement) happens at the coordinator level (i.e., this `/develop` command).

```markdown
# /project:develop — Full Development Pipeline

You are a seasoned Project Lead. Orchestrate the development of a new feature
or bug fix from conception to delivery. Follow each phase rigorously.

## Phase 1: UNDERSTAND & SPECIFY

1. Spawn @architect:
   "@architect Conduct a spec interview with the user for the feature: $ARGUMENTS.
   Write the confirmed spec to `docs/SPRINT.md`. Do NOT start coding."

2. **User Interaction:** Present the generated `docs/SPRINT.md` to the user. Use /clarify
   to ask for approval or revisions. Do NOT proceed until the spec is approved.

## Phase 2: PLAN & REFINE

1. Spawn @planner:
   "@planner Read `docs/SPRINT.md` and the existing codebase. Create a step-by-step
   implementation plan for the approved spec. Consider existing patterns,
   dependencies, and potential risks. Include testing strategy and expected outputs.
   Write the plan into a new markdown file `docs/PLAN.md`."

2. Spawn @grill (on plan):
   "@grill Conduct an adversarial review of the `docs/PLAN.md` file (the implementation
   plan, NOT code). Identify missing steps, architectural flaws, unaddressed risks,
   and provide concrete action items. Output findings to `docs/PLAN_REVIEW.md`."

3. **User Interaction:** Present `docs/PLAN_REVIEW.md` to the user. Use /clarify to ask
   for approval to refine the plan based on feedback, or to ignore feedback. If
   refinement is chosen, loop back to Step 1 in this phase with revised instructions
   for @planner and @grill. Do NOT proceed until the plan is approved.

## Phase 3: IMPLEMENT (TDD-driven)

1. Create a isolated git worktree for the feature: `!git worktree add ../feature-branch -b feature/$ARGUMENTS`
   All subsequent implementation steps MUST occur within this worktree.

2. For each major task or logical chunk in `docs/PLAN.md`:
   a. Spawn @implementer (TDD focus):
      "@implementer Implement section N of the plan from `docs/PLAN.md` using a
      test-driven development approach. First, write a failing test. Then, write
      the minimum code to make it pass. Finally, refactor. Commit intermediate
      steps if necessary. Provide commit hashes."

   b. Immediately after each implementation segment, spawn @verifier:
      "@verifier Rigorously verify the latest changes by @implementer in the
      worktree. Run build, run tests, run linters. Perform at least one adversarial
      probe (e.g., boundary values, idempotency, concurrency). Report `VERDICT: PASS/FAIL/PARTIAL`."

   c. If @verifier reports `FAIL` or `PARTIAL`:
      Spawn @fixer:
      "@fixer Address the issues identified by @verifier in `docs/VERIFIER_REPORT.md`.
      Fix the code, ensure all tests pass, and resolve any build/lint errors.
      Commit the fix."
      Loop back to (b) for reverification until `VERDICT: PASS`.

## Phase 4: FINAL REVIEW & DELIVER

1. Spawn @grill (on code):
   "@grill Conduct a comprehensive adversarial code review of ALL changes in the
   current worktree. Focus on security, performance, maintainability, and architectural
   soundness. Output BLOCKERS, SHOULD-FIX, NITPICKs with suggested alternatives to `docs/CODE_REVIEW.md`."

2. **User Interaction:** Present `docs/CODE_REVIEW.md` to the user via /clarify. Address
   all BLOCKERs by spawning @fixer again until resolved. Confirm readiness to deliver.

3. **Automated Documentation Updates:**
   Spawn @doc-updater (NEW AGENT):
   "@doc-updater Review `docs/SPRINT.md`, `docs/PLAN.md`, `docs/PLAN_REVIEW.md`,
   `docs/CODE_REVIEW.md`, and the conversation history for key decisions and learnings.
   Extract new architectural decisions and add to `docs/DECISIONS.md`.
   Extract non-obvious discoveries, major bug fixes, or new patterns and add to `docs/LEARNINGS.md`.
   Ensure `docs/SPRINT.md`, `docs/PLAN.md`, `docs/PLAN_REVIEW.md`, and `docs/CODE_REVIEW.md` are archived or cleaned up."

4. Commit all remaining changes, including docs updates: `!git add -A && git commit -m "feat($ARGUMENTS): ship feature $ARGUMENTS"`

5. Publish to `npm` (if applicable) and open pull request:
   `!npm version patch && npm publish`
   `!gh pr create --title "feat($ARGUMENTS): $ARGUMENTS" --body "$ARGUMENTS shipped via /develop pipeline"`
   `!git worktree remove ../feature-branch`

```

**Generated files:**
- `.claude/commands/project-develop.md`

## Layer 2: Building Blocks (Commands & Agents)

These commands and agents, previously defined, serve as the components orchestrated by `/project:develop`.

### Commands

- `/project:spec` (now orchestrated by @architect, output `docs/SPRINT.md`)
- `/project:plan` (now orchestrated by @planner, output `docs/PLAN.md`)
- `/project:grill` (now orchestrated by @grill, output `docs/PLAN_REVIEW.md` or `docs/CODE_REVIEW.md`)
- `/project:prove` (functionality covered by @verifier in /develop)
- `/project:review` (now orchestrated by @grill for final code review)
- `/project:commit` (integrated into Phase 5: DELIVER of /develop)
- `/project:fix` (used by /develop when @verifier fails)

### Agents

- `@explorer` (used in /project:ultraplan, can be adapted for @planner to explore)
- `@verifier` (key role in Phase 3 & 4 of /develop)
- `@architect` (key role in Phase 1 of /develop) — *New agent derived from agent-prompt-agent-creation-architect.md*
- `@planner` (key role in Phase 2 of /develop) — *New agent derived from agent-prompt-plan-mode-enhanced.md*
- `@implementer` (key role in Phase 3 of /develop) — *New agent; will need a new prompt for TDD-focused implementation*
- `@fixer` (key role in Phase 3 & 4 of /develop) — *New agent; will need a new prompt for targeted bug fixing*
- `@doc-updater` (NEW AGENT, key role in Phase 5 of /develop) — *New prompt for automating `DECISIONS.md` and `LEARNINGS.md` updates*
- `@security-monitor` (integration for Level 3-4 environments)
- `@worker` (for nested parallel tasks within /develop, e.g. /batch)

**Note:** Existing agents (`@qa-orchestrator`, `@linter`, `@e2e-tester`) will be integrated into `@verifier`'s workflow.

## Layer 3: Engineering Principles (CLAUDE.md Directives)

These general directives apply across all agent interactions and reinforce best practices.

### Output Efficiency
```markdown
## Engineering Standards
- Lead with answers over reasoning. Be concise.
- Use absolute file paths in all references.
- No filler, no inner monologue, no time estimates.
- Produce "load-bearing code snippets" — every line of output should be actionable.
```

### Tool Usage Policy  
```markdown
## Tool Usage Policy
- Prefer Edit tool over sed/awk for file modifications
- Prefer Grep tool over rg for searching
- Prefer Read tool over cat for file reading
- Prefer Write tool over echo/heredoc for file creation
- Reserve Bash for: builds, installs, git, network, processes
- Prefer editing existing files over creating new ones
- Delete unused code completely — no compatibility shims
- Read and understand existing code before modifying
```

### Git Safety
```markdown
## Git Safety
- Never use `git add .` — stage specific files
- Never amend commits unless explicitly requested
- Never skip pre-commit hooks unless explicitly requested
- Prefer new commits over amending
- Use conventional commit format
- Small, atomic commits (one logical change per commit)
```

### No Premature Abstractions
```markdown
## Code Philosophy
- Do not create abstractions for one-time operations or hypothetical requirements
- Complete the task fully — don't gold-plate, but don't leave it half-done
- Prefer editing existing files over creating new ones
- Delete unused code completely rather than adding compatibility shims
```

### Coordinator Precision (for Level 3-4 environments)
```markdown
## Multi-Agent Communication
- When delegating to subagents, provide specific file paths and line numbers
- Never use vague instructions like "based on your findings"
- Each worker prompt must be fully self-contained — workers cannot ask questions
- Include: goal, specific task, codebase conventions, test recipe
```

### Deferred Tool Discovery
```markdown
## Context Management
- Start with core tools only. Search for specialized tools when needed.
- If context is getting long, summarize completed work before continuing.
- When exploring, use parallel tool calls for efficiency.
```

## Documentation Simplification & Automation

**Problem:** `TODO.md`, `DECISIONS.md`, `LEARNINGS.md`, `SPRINT.md` are often stale or unused for Kairn's own development.

**Solution:** Make these files an active part of the `/project:develop` workflow.

1.  **Kill `TODO.md`:** `ROADMAP.md` is the source of truth for high-level tasks. `docs/SPRINT.md` will become the living, short-term task list.
    *   **Action:** Remove `TODO.md` from the generated environment for Kairn. Ensure old `docs/TODO.md` is deleted.

2.  **`docs/SPRINT.md` as Living Spec/Plan:** It becomes the primary scratchpad for `/project:spec` and `/project:plan`, updated directly by agents.
    *   **Action:** Ensure @architect and @planner agents write to and read from `docs/SPRINT.md`.

3.  **Automated `DECISIONS.md` and `LEARNINGS.md`:** Integrate updates into Phase 5 (DELIVER) of `/project:develop` via the new `@doc-updater` agent.
    *   **Action:** Create `@doc-updater` agent. Include a post-completion hook in `/project:develop` to invoke `@doc-updater`.

## What We're NOT Doing (Updated)

1.  **Not directly mapping /ultraplan to /develop:** Instead, `/develop` is Kairn's own version of advanced multi-agent orchestration, inspired by /ultraplan.
2.  **Not explicitly adding /schedule:** That requires Anthropic's cloud CCR infrastructure, which is beyond the scope of a local CLI.
3.  **Not adding dream memory consolidation:** That's Claude Code's internal memory system, not harness-level.
4.  **Still killing `TODO.md`** and consolidating short-term tasks into `docs/SPRINT.md` if the user wants it.
