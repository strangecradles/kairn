## 1. Verification Section in CLAUDE.md Template

### Background
Boris Cherny's #1 tip: "Give Claude a way to verify its output. Once you do that, Claude will iterate until the result is correct."

### Problem
Our current CLAUDE.md template has no explicit verification section. Claude doesn't know how to check its own work for a given project.

### Solution
Add a mandatory "Verification" section to the CLAUDE.md template in `src/compiler/prompt.ts`:

```markdown
## Verification
After implementing any change, verify it works:
- {build command} — must pass with no errors
- {test command} — all tests must pass
- {lint command} — no warnings or errors
- {type check command} — no type errors

If any verification step fails, fix the issue before moving on.
Do NOT skip verification steps.
```

### Implementation
- Update `src/compiler/prompt.ts` to add "Verification" as the 8th required section in the CLAUDE.md template
- The LLM should populate verification commands based on detected tech stack (e.g., `npm run build`, `pytest`, `cargo test`)
- For research/writing projects, verification looks different: "Verify all sources in docs/SOURCES.md are cited in the summary"
- Update `src/templates/claude-md.md` with the new section

## 2. Known Gotchas Section in CLAUDE.md Template

### Background
Boris's habit: after every correction, tell Claude "Update your CLAUDE.md so you don't make that mistake again." Then prune periodically.

### Problem
Our generated CLAUDE.md is static. There's no designated place for project-specific mistakes to accumulate.

### Solution
Add a "Known Gotchas" section to every generated CLAUDE.md:

```markdown
## Known Gotchas

- (none yet — this section grows as you work)
```

### Implementation
- Update `src/compiler/prompt.ts` to include this section in the template
- Update `src/templates/claude-md.md`
- The section starts empty with instructions in HTML comments (so Claude sees them but they don't clutter the output)

## 3. `/project:spec` Command — Interview-Based Spec Creation

### Background
Boris recommends: "Start with a minimal spec and ask Claude to interview you using the AskUserQuestion tool to refine it."

### Problem
Our `/project:sprint` command defines acceptance criteria, but there's no interview step. The user has to know what to specify upfront.

### Solution
Add a `/project:spec` command that interviews the user before coding:

```markdown
# .claude/commands/spec.md
Before building this feature, interview me to create a complete spec.

Ask me 5-8 questions, one at a time:
1. What specifically should this feature do? (the "what")
2. Who uses it and how? (the "who")
3. What are the edge cases or error states?
4. How will we know it works? (acceptance criteria)
5. What should it explicitly NOT do? (scope boundaries)
6. Any dependencies, APIs, or constraints?
7. How does it fit with existing code? (integration points)
8. What's the priority — speed, quality, or flexibility?

After my answers, write a structured spec to docs/SPRINT.md:
- Feature name
- Description (from my answers, not invented)
- Acceptance criteria (testable)
- Out of scope
- Technical approach (suggested)

Do NOT start coding until I confirm the spec.
```

### Implementation
- Add to the "always include for code projects" list in `src/compiler/prompt.ts`
- Add template to `src/templates/commands.md`

## 4. `/project:prove` Command — Verification on Demand

### Background
Boris's pattern: "Prove to me this works. Compare behavior on main vs this branch (tests/logs/output diffs)."

### Solution
```markdown
# .claude/commands/prove.md
Prove the current implementation works.

1. Run the full test suite:
!{TEST_COMMAND} 2>&1

2. Compare against the main branch:
!git diff main --stat 2>/dev/null

3. Show specific evidence:
   - Test results (pass/fail counts)
   - Behavioral diff (output on main vs this branch)
   - Edge cases tested
   - Error handling verified

4. Rate your confidence:
   - HIGH: All tests pass, edge cases covered, no regressions
   - MEDIUM: Core functionality works, some edge cases untested
   - LOW: Needs more verification before shipping

If confidence is LOW or MEDIUM, explain what's missing and fix it.
```

### Implementation
- Add to code project command set in `src/compiler/prompt.ts`
- Add template to `src/templates/commands.md`

## 5. `/project:grill` Command — Adversarial Review

### Background
Boris's pattern: "Grill me on these changes and don't make a PR until I pass your test."

### Solution
```markdown
# .claude/commands/grill.md
Review the current changes adversarially.

!git diff --staged 2>/dev/null || git diff HEAD 2>/dev/null

Act as a senior engineer doing a thorough code review.
For each file changed, challenge me:

1. "Why did you choose this approach over X?"
2. "What happens if Y input is provided?"
3. "Have you considered the performance impact of Z?"
4. "This could break if..."

Be tough but specific. For each concern:
- Severity: BLOCKER / SHOULD-FIX / NITPICK
- The exact scenario that could fail
- A suggested alternative

Do NOT approve until all BLOCKERs are resolved.
```

### Implementation
- Add to code project command set in `src/compiler/prompt.ts`
- Add template to `src/templates/commands.md`

## 6. `/project:reset` Command — Clean Restart

### Background
Boris's pattern: "Knowing everything you know now, scrap this and implement the elegant solution."

### Solution
```markdown
# .claude/commands/reset.md
Stop. Read docs/DECISIONS.md and docs/LEARNINGS.md.

Considering everything we've learned so far:
1. What was the original approach?
2. What went wrong or feels inelegant?
3. What would the clean, from-scratch solution look like?

Propose the new approach. Do NOT implement yet.
If I approve, stash the current changes and start fresh:
  git stash -m "pre-reset: $(date +%Y%m%d-%H%M)"

Then implement the elegant solution.
```

### Implementation
- Add to code project command set in `src/compiler/prompt.ts`
- Add template to `src/templates/commands.md`

## 7. Statusline Configuration

### Background
Boris and team customize Claude Code's statusline to show context usage, git branch, and session cost.

### Solution
Generate a statusline config in settings.json for code projects:

```json
{
  "statusLine": {
    "command": "printf '%s | %s tasks' \"$(git branch --show-current 2>/dev/null || echo 'no-git')\" \"$(grep -c '\\- \\[ \\]' docs/TODO.md 2>/dev/null || echo 0)\""
  }
}
```

### Implementation
- Update `src/adapter/claude-code.ts` to include statusLine in generated settings.json
- Only include for code projects (need git)
- Scanner should detect if git is initialized
