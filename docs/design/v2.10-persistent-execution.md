# v2.10.0 — Persistent Execution Loop

> Inspired by OMC's Ralph loop. The single biggest gap between Kairn-generated harnesses and production agent workflows: persistence. Today, `/project:develop` runs once linearly. If it fails, the user restarts manually. A persistence loop keeps working until all acceptance criteria pass, tracks progress across retries, and auto-routes complex tasks through the loop.

## Problem

Kairn-generated harnesses have a linear execution model:

1. **`/project:develop` runs once.** It orchestrates @architect → @planner → @implementer → @verifier → @fixer → @grill → @doc-updater through seven phases. If Phase 4 (Verify) fails, Phase 5 (@fixer) attempts one round of fixes, then Phase 6 (@grill) reviews. But if the grill finds blockers, the pipeline stops. The user must manually restart, losing context about what was already tried.

2. **No structured progress tracking.** The only record of what's done lives in git commits and `docs/SPRINT.md` status. There's no machine-readable state that a resumed session can parse to know "criteria 1-3 passed, criterion 4 failed on retry 2, criteria 5-6 haven't been attempted."

3. **No auto-routing for complex tasks.** When a user types "add OAuth authentication with Google and GitHub providers," that's clearly a multi-step task that benefits from persistent tracking. But the harness treats it the same as "fix the typo in README" — both get one-shot execution unless the user explicitly invokes `/project:develop`.

4. **Session boundaries break execution.** v2.9.0 added `memory.json` persistence for decisions and sprint status, but there's no structured resume protocol. If a session ends mid-implementation, the next session sees the memory context but doesn't know what step to resume from.

**What OMC's Ralph loop gets right:**
- Story-by-story acceptance tracking (`prd.json` with per-story status)
- Retry-until-done iteration with configurable max retries
- Reviewer verification gate before marking complete
- Session-persistent state files that survive restarts

**What we distill (not copy):**
- OMC manages 150 skills, tmux workers, ultrawork parallelism, and a plugin system. Most projects don't need that complexity.
- We extract the **persistence pattern** — structured progress, retry loops, reviewer gates, session resume — and embed it in the harness as a single command template + routing hook.

## Solution

### 1. `/project:persist` Command Template

A new command generated in every code project harness (alongside `/project:develop`, `/project:sprint`, etc.). It reads acceptance criteria from `docs/SPRINT.md`, works through them story-by-story, tracks progress in `.claude/progress.json`, and loops until all criteria pass or max retries are exhausted.

```markdown
# Persistent Execution Loop

Read acceptance criteria from docs/SPRINT.md. Work through each criterion
until ALL pass, tracking progress in .claude/progress.json.

## Protocol

1. **Load or Initialize Progress**
   Read .claude/progress.json if it exists (resume). Otherwise, parse
   docs/SPRINT.md for numbered acceptance criteria and initialize progress.

2. **For Each Incomplete Criterion (in order):**
   a. Read the criterion from docs/SPRINT.md
   b. Check if prior attempts exist in progress.json — read failure notes
   c. Implement or fix the criterion
   d. Run verification: build, test, typecheck, lint
   e. If verification passes → mark PASSED in progress.json, record evidence
   f. If verification fails → increment retry count, record failure reason
      - If retries < 3: loop back to (c) with failure context
      - If retries >= 3: mark BLOCKED, move to next criterion
   g. Update .claude/progress.json after each criterion attempt

3. **After All Criteria Attempted:**
   - If any BLOCKED: report which criteria are blocked and why
   - If all PASSED: proceed to review gate

4. **Review Gate**
   Delegate to @grill agent for adversarial review of all changes.
   - If @grill returns BLOCKERs: fix each blocker, re-verify affected criteria
   - If @grill returns PASS: mark sprint as COMPLETE in progress.json

5. **Persist State**
   Write final .claude/progress.json. If session is ending, ensure
   memory.json includes progress summary for next session pickup.

## Resume Protocol

When progress.json exists with incomplete criteria:
- Skip criteria already marked PASSED (unless their files were modified since)
- Resume from the first non-PASSED criterion
- Carry forward failure notes from prior attempts

## Constraints

- Maximum 3 retries per criterion before marking BLOCKED
- Maximum 1 review gate cycle (fix blockers once, re-grill once)
- Always run full verification (build + test + typecheck + lint) per criterion
- Never modify criteria in SPRINT.md — only progress.json tracks status
```

### 2. `progress.json` Schema

Machine-readable execution state, stored at `.claude/progress.json`. Survives session boundaries. The persist command reads this on startup to resume, and writes it after every criterion attempt.

```typescript
interface ProgressFile {
  /** Schema version for forward compatibility */
  version: 1;

  /** Source of acceptance criteria */
  sprintFile: string;  // "docs/SPRINT.md"

  /** When the persistence loop was first started */
  startedAt: string;   // ISO 8601

  /** When last updated */
  updatedAt: string;   // ISO 8601

  /** Overall sprint status */
  status: 'in_progress' | 'review' | 'complete' | 'blocked';

  /** Per-criterion tracking, keyed by criterion number */
  criteria: Record<string, CriterionProgress>;

  /** Review gate results */
  review?: ReviewGate;

  /** Session history — breadcrumb for resume */
  sessions: SessionEntry[];
}

interface CriterionProgress {
  /** The criterion number from SPRINT.md (e.g., "1", "2") */
  id: string;

  /** Short description extracted from SPRINT.md */
  description: string;

  /** Current status */
  status: 'pending' | 'in_progress' | 'passed' | 'blocked';

  /** Number of implementation attempts */
  attempts: number;

  /** Max retries before marking blocked */
  maxAttempts: 3;

  /** History of each attempt */
  history: AttemptRecord[];

  /** Evidence of passing (test output, build log summary) */
  evidence?: string;

  /** Git commit SHA when this criterion was completed */
  commitSha?: string;
}

interface AttemptRecord {
  /** Attempt number (1-indexed) */
  attempt: number;

  /** When this attempt started */
  startedAt: string;

  /** When this attempt ended */
  completedAt: string;

  /** What happened */
  outcome: 'passed' | 'failed';

  /** What failed and why (for failed attempts) */
  failureReason?: string;

  /** Which verification step failed: build, test, typecheck, lint */
  failedStep?: 'build' | 'test' | 'typecheck' | 'lint';

  /** Files modified during this attempt */
  filesModified: string[];
}

interface ReviewGate {
  /** When the review was conducted */
  reviewedAt: string;

  /** Review verdict */
  verdict: 'pass' | 'blockers' | 'should_fix';

  /** Blocker descriptions (if any) */
  blockers: string[];

  /** Whether blockers were resolved in a fix round */
  blockersResolved: boolean;
}

interface SessionEntry {
  /** Session start time */
  startedAt: string;

  /** What was the state when this session started */
  resumedFrom: 'fresh' | 'in_progress' | 'review';

  /** Which criteria were worked on */
  criteriaAttempted: string[];

  /** Summary of session outcome */
  summary: string;
}
```

**Design decisions:**

- **Keyed by criterion number, not description.** Criterion text may get lightly edited in SPRINT.md; the number is stable.
- **Attempt history is append-only.** Never overwrite prior attempts — the persist command reads failure reasons from prior attempts to avoid repeating the same approach.
- **Sessions array is a breadcrumb.** When a new session starts and finds progress.json, it adds a SessionEntry with `resumedFrom` to show the resume chain.
- **No nested story/epic hierarchy.** SPRINT.md criteria are flat and numbered. If a criterion is too large, the user should break it into sub-criteria in SPRINT.md, and progress.json will track each independently.
- **Version field for forward compatibility.** If v2.11.0 extends the schema, the persist command can detect v1 files and migrate.

### 3. `UserPromptSubmit` Auto-Routing Hook

A `UserPromptSubmit` hook that analyzes incoming prompts for complexity signals and routes matching tasks through `/project:persist` instead of one-shot execution.

#### Hook Implementation (generated in `.claude/hooks/persist-router.mjs`)

```javascript
import { readFileSync } from 'fs';

/**
 * Analyze prompt complexity and route to /project:persist when appropriate.
 *
 * Complexity signals (any 2+ triggers routing):
 * - Multi-step: "then", "after that", "and also", numbered steps
 * - Feature scope: "add", "implement", "build", "create" + noun phrases
 * - Refactor scope: "migrate", "convert", "replace X with Y", "upgrade"
 * - Bug with repro: "when X happens", "steps to reproduce", "broken"
 * - Explicit: "persist", "keep working", "don't stop"
 *
 * Pass-through (no routing):
 * - Questions: starts with "what", "how", "why", "where", "can you explain"
 * - Single file: "edit foo.ts", "fix the typo in", "update the comment"
 * - Lookups: "show me", "find", "search for", "list"
 * - Commands: starts with "/project:" (user is already using a command)
 */
const input = JSON.parse(readFileSync('/dev/stdin', 'utf8'));
const prompt = (input.prompt ?? '').trim();

// Pass-through patterns (fast exit)
const PASSTHROUGH = /^(what|how|why|where|when|can you|does|is |show me|find |search |list |\/project:)/i;
const SINGLE_FILE = /^(edit|fix the typo|update the comment|change the|rename) .{3,60}$/i;

if (PASSTHROUGH.test(prompt) || SINGLE_FILE.test(prompt) || prompt.length < 20) {
  process.stdout.write(JSON.stringify({ continue: true }));
  process.exit(0);
}

// Check config for routing mode
let routingMode = 'auto';
try {
  const settings = JSON.parse(readFileSync('.claude/settings.json', 'utf8'));
  routingMode = settings.persistence_routing ?? 'auto';
} catch { /* default to auto */ }

if (routingMode === 'off') {
  process.stdout.write(JSON.stringify({ continue: true }));
  process.exit(0);
}

// Complexity signals
const signals = [];

// Multi-step indicators
if (/\b(then|after that|and also|next|finally|step \d|first .* then)\b/i.test(prompt)) {
  signals.push('multi-step');
}

// Feature-scale work
if (/\b(add|implement|build|create|integrate|set up)\b.*\b(feature|auth|api|endpoint|page|component|module|service|database|migration)\b/i.test(prompt)) {
  signals.push('feature-scope');
}

// Refactor-scale work
if (/\b(migrate|convert|replace|upgrade|refactor|rewrite|restructure)\b/i.test(prompt)) {
  signals.push('refactor-scope');
}

// Bug with reproduction context
if (/\b(when .* happens|steps to reproduce|broken|crash|regression|fails when)\b/i.test(prompt)) {
  signals.push('bug-with-repro');
}

// Explicit persistence request
if (/\b(persist|keep working|don't stop|until done|until .* pass)\b/i.test(prompt)) {
  signals.push('explicit');
}

// Word count as a complexity proxy (long prompts = complex tasks)
if (prompt.split(/\s+/).length > 50) {
  signals.push('long-prompt');
}

const shouldRoute = routingMode === 'manual'
  ? signals.includes('explicit')
  : signals.length >= 2 || signals.includes('explicit');

if (shouldRoute) {
  // Route through persist — inject the command prefix
  process.stdout.write(JSON.stringify({
    continue: true,
    hookSpecificOutput: {
      hookEventName: 'UserPromptSubmit',
      additionalContext: [
        'PERSISTENCE ROUTING: This task has complexity signals (' + signals.join(', ') + ').',
        'Execute this using the /project:persist workflow:',
        '1. Ensure acceptance criteria exist in docs/SPRINT.md (create from this prompt if needed)',
        '2. Initialize .claude/progress.json',
        '3. Work criterion-by-criterion until all pass',
        '4. Run review gate before marking complete',
      ].join('\n'),
    },
  }));
} else {
  process.stdout.write(JSON.stringify({ continue: true }));
}
```

#### Configuration

The routing hook respects a `persistence_routing` field in `settings.json`:

| Value | Behavior |
|-------|----------|
| `auto` (default) | Route when 2+ complexity signals detected |
| `manual` | Route only on explicit "persist" / "keep working" / "until done" keywords |
| `off` | Never auto-route; user must invoke `/project:persist` directly |

This is set during compilation based on autonomy level:
- **Level 1-2:** `manual` — persistence only when explicitly requested
- **Level 3:** `auto` — automatic routing for complex tasks
- **Level 4:** `auto` — automatic routing (full auto already implies persistence)

### 4. Integration with Existing Systems

#### 4a. Sprint Contract Integration

The persist command depends on `docs/SPRINT.md` containing numbered acceptance criteria — the exact format that `/project:sprint` and `@architect` already produce (added in v2.9.0).

**Flow:**
```
/project:sprint  →  docs/SPRINT.md (numbered criteria)
                          ↓
/project:persist  →  reads SPRINT.md, initializes progress.json
                          ↓
                     works criterion-by-criterion
                          ↓
                     @grill review gate
                          ↓
                     progress.json status: complete
```

If `/project:persist` is invoked without existing criteria in SPRINT.md, it first delegates to `@architect` to extract criteria from the user's prompt, writes them to SPRINT.md, then begins the loop. This makes the routing hook work even for free-form prompts that haven't gone through `/project:sprint`.

**No changes to SPRINT.md format.** The persist command reads the existing numbered criteria format:
```markdown
## Acceptance Criteria
1. User can sign in with Google OAuth
2. User can sign in with GitHub OAuth
3. Session tokens stored securely in httpOnly cookies
...
```

Progress tracking lives exclusively in `progress.json` — SPRINT.md remains the source of truth for *what* to build, progress.json tracks *execution state*.

#### 4b. Memory Persistence Integration

v2.9.0 added `SessionEnd`/`SessionStart` hooks that save/load `.claude/memory.json`. The persist command integrates with this in two ways:

**SessionEnd enhancement:** When `progress.json` exists with status `in_progress`, the SessionEnd hook includes a progress summary in `memory.json`:

```json
{
  "decisions": "...",
  "sprint": "...",
  "gotchas": "...",
  "persistence": {
    "active": true,
    "criteriaTotal": 5,
    "criteriaPassed": 3,
    "criteriaBlocked": 0,
    "nextCriterion": "4",
    "lastAttemptSummary": "Criterion 3 passed on attempt 2 after fixing import path"
  },
  "timestamp": "2026-04-02T..."
}
```

**SessionStart enhancement:** When `memory.json` contains `persistence.active: true`, the SessionStart hook injects a resume prompt:

```
RESTORED SESSION MEMORY:
...existing memory context...

ACTIVE PERSISTENCE LOOP: 3/5 criteria passed. Resume with /project:persist
or continue manually. Progress tracked in .claude/progress.json.
```

This gives the agent (or user) a clear signal that there's an in-progress persistence loop to resume, without forcing automatic resumption (which could be surprising).

**Implementation in HARNESS_PROMPT:** The SessionEnd hook template gains a conditional block that checks for `.claude/progress.json` and appends the persistence summary to the memory payload. The SessionStart hook template gains a conditional block that detects the persistence field and adds the resume prompt.

#### 4c. `/project:develop` Relationship

`/project:persist` does NOT replace `/project:develop`. They serve different purposes:

| | `/project:develop` | `/project:persist` |
|---|---|---|
| **When** | Single-session feature implementation | Multi-criterion, potentially multi-session work |
| **Tracking** | Linear phase progression (1-7) | Per-criterion progress with retries |
| **Resume** | No (restarts from Phase 1) | Yes (reads progress.json) |
| **Routing** | Manual invocation | Auto-routed for complex tasks |
| **Review** | Phase 6 (@grill) | Review gate after all criteria pass |

For simple features that fit in one session, `/project:develop` is still the right tool. `/project:persist` is for the 20% of tasks where you need retry resilience and session persistence.

The persist command can internally delegate to `/project:develop`-style phases for each criterion — it orchestrates at the criterion level, while develop orchestrates at the phase level within a criterion.

### 5. Eval Template: `persistence-completion`

A new eval template that tests whether the persistence loop actually completes multi-step tasks. This is critical — without a measurable signal, we can't know if `/project:persist` improves outcomes vs. one-shot execution.

#### Template Definition (added to `src/evolve/templates.ts`)

```typescript
'persistence-completion': {
  id: 'persistence-completion',
  name: 'Persistence Completion',
  description: 'Can the agent complete a multi-criterion task using the persistence loop?',
  bestFor: ['feature-development', 'full-stack', 'api-building', 'maintenance'],
}
```

#### Task Generation Guidance (added to `TASK_GENERATION_PROMPT`)

```
- persistence-completion: Task MUST have 3+ acceptance criteria that require
  sequential implementation (later criteria depend on earlier ones). The task
  description should be a realistic feature request, NOT broken into steps
  by the eval — the agent must parse it into criteria via /project:sprint or
  the persist command's auto-extraction. Judge by:
  (a) Were all criteria met? Check progress.json for status: complete
  (b) Did the agent use structured progress tracking? Check .claude/progress.json exists
  (c) Did retries improve outcomes? If a criterion failed then passed, that's signal
  (d) Was a review gate run? Check progress.json review field
```

#### Example Generated Task

```yaml
- id: add-user-preferences-api
  template: persistence-completion
  description: >
    Add a user preferences API. Users should be able to:
    1. GET /api/preferences/:userId — returns saved preferences
    2. PUT /api/preferences/:userId — updates preferences (validate schema)
    3. Preferences should have defaults when a user has none saved
    4. Add integration tests for all three behaviors
    5. Update API docs in docs/API.md
  setup: "npm install"
  expected_outcome: |
    - All 5 acceptance criteria pass verification
    - .claude/progress.json exists with status: complete
    - progress.json shows all criteria as "passed"
    - Tests pass: npm test includes preference API tests
    - API docs updated
  scoring: rubric
  rubric:
    - criterion: "All acceptance criteria met"
      weight: 40
      check: "cat .claude/progress.json | jq '.status == \"complete\"'"
    - criterion: "Structured progress tracking used"
      weight: 20
      check: "test -f .claude/progress.json && cat .claude/progress.json | jq '.criteria | length >= 3'"
    - criterion: "Tests pass"
      weight: 25
      check: "npm test 2>&1 | grep -q 'Tests.*passed\\|0 failed'"
    - criterion: "Review gate executed"
      weight: 15
      check: "cat .claude/progress.json | jq '.review != null'"
  timeout: 900
```

## Non-Goals

- **No runtime daemon or background process.** The persist command is a prompt template, not a persistent service. It runs within a Claude Code session.
- **No parallel criterion execution.** Criteria are worked sequentially — later criteria often depend on earlier ones. Parallelism is a v2.11.0 optimization.
- **No automatic session resumption.** The SessionStart hook *informs* the agent about active persistence loops, but doesn't auto-invoke `/project:persist`. The user decides whether to resume.
- **Not copying OMC's plugin system, tmux workers, or ultrawork parallelism.** We extract the persistence pattern only.

## Risk Assessment

| Risk | Mitigation |
|------|-----------|
| Auto-routing false positives (simple tasks routed through persist) | Pass-through patterns are checked first; `manual` mode available; 2-signal threshold for `auto` mode |
| progress.json becomes stale after manual edits | Resume protocol checks file modification times against commit SHAs; stale evidence triggers re-verification |
| Infinite retry loops | Hard cap: 3 retries per criterion, 1 review fix cycle, then BLOCKED |
| Context window bloat from progress tracking | progress.json is written to disk, not injected into context; only the summary (which criterion is next) is injected via memory.json |
| Conflict between persist and develop commands | Clear documentation: develop = single-session linear pipeline, persist = multi-criterion retry loop. They can compose (persist calls develop-style phases per criterion) |
