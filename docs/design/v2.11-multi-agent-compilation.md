# v2.11.0 — Multi-Agent Compilation Pipeline

> From monolithic LLM call to orchestrated specialist agents producing typed HarnessIR nodes.

**Status:** Design  
**Depends on:** v2.10.0 (persistent execution), v2.7.0 (HarnessIR)  
**Fixes:** JSON truncation in Pass 2 (#inferix-bug), harness quality ceiling from single-agent generation

---

## Problem Statement

### The Truncation Bug

Pass 2 of compilation asks a single LLM to produce a ~16K token JSON blob containing all harness content: CLAUDE.md (~150 lines), 20 commands, 11 agents, rules, docs, skills, and hooks — all JSON-escaped strings inside one object. With `maxTokens: 8192`, the model hits the limit mid-string, producing truncated JSON:

```
✖ Compilation failed
Failed to parse harness JSON: Unterminated string in JSON at position 25629
```

The "concise retry" only constrains CLAUDE.md and commands, but agents/skills/docs are still large. Even bumping to `maxTokens: 16384` is a bandaid — complex workflows can exceed any static limit.

### The Quality Ceiling

A single model producing 7 different artifact types (sections, commands, agents, rules, skills, docs, hooks) in one shot is a blunt object. Each artifact type has distinct conventions:

| Artifact | Convention | Format |
|----------|-----------|--------|
| CLAUDE.md sections | Structured markdown, mandatory template | Heading + paragraphs |
| Commands | `!` shell integration, `$ARGUMENTS`, orchestration prompts | Markdown task instructions |
| Agents | YAML frontmatter (`modelRouting`, `disallowedTools`), persona | Persona + delegation rules |
| Rules | Path-scoped YAML frontmatter, constraint bullets | Scoped instructions |
| Docs | Template structures, acceptance criteria format | Structured templates |
| Skills | SKILL.md format, 3-phase patterns | Procedural guides |

One system prompt cannot specialize in all six formats simultaneously.

### The Architecture Gap

Compilation produces `EnvironmentSpec` (flat JSON with string values). Evolution works on `HarnessIR` (typed nodes: `CommandNode[]`, `AgentNode[]`, etc.). These don't share types. Compilation → adapter → files → parse → `HarnessIR` is a lossy round-trip through string serialization.

**Compilation should produce HarnessIR directly**, aligning both pipelines around the same representation.

## Design

### Architecture

```
Pass 1: Skeleton (unchanged)
    │
    ▼
Pass 2: @orchestrator (LLM)
    │  Reads skeleton + intent
    │  Emits CompilationPlan: what to generate, in what order
    │
    ▼
Pass 3: Specialist agents (parallel with dependencies)
    │
    │  Phase A (no dependencies, parallel):
    │  ├── @sections-writer → Section[]
    │  ├── @rule-writer     → RuleNode[]
    │  └── @doc-writer      → DocNode[]
    │
    │  Phase B (depends on Phase A, parallel):
    │  ├── @command-writer  → CommandNode[]  (sees sections for context)
    │  └── @agent-writer    → AgentNode[]    (sees rules for scope)
    │
    │  Phase C (depends on Phase B, serial):
    │  └── @linker          → validates cross-references, patches broken refs
    │
    ▼
Pass 4: Deterministic assembly (unchanged, from current Pass 3)
    │  settings.json, .mcp.json, intent patterns, hooks
    │
    ▼
HarnessIR → adapter → files
```

### The @orchestrator Agent

The orchestrator is the only agent that sees the full picture. Given the skeleton (tool selections + project outline) and user intent, it produces a `CompilationPlan`:

```typescript
interface CompilationPlan {
  project_context: string;        // 2-3 sentence project summary for downstream agents
  phases: CompilationPhase[];     // topologically sorted
}

interface CompilationPhase {
  id: string;                     // "phase-a", "phase-b", "phase-c"
  agents: AgentTask[];
  dependsOn: string[];            // phase IDs that must complete first
}

interface AgentTask {
  agent: 'sections-writer' | 'command-writer' | 'agent-writer' | 'rule-writer' | 'doc-writer' | 'skill-writer';
  items: string[];                // what to generate: ["purpose", "tech-stack", "commands", ...]
  context_hint?: string;          // agent-specific guidance from orchestrator
  max_tokens: number;             // budget for this agent
}
```

**Why LLM orchestrator instead of deterministic?** The skeleton already contains `key_commands`, `custom_agents`, etc. — but an LLM orchestrator can:
1. Decide command granularity (simple project → 8 commands; complex → 20)
2. Detect when custom agents are needed beyond the standard set
3. Allocate token budgets based on project complexity
4. Express inter-agent dependencies the skeleton can't capture
5. **Scale to swarms**: when compiling for 200 agents, the orchestrator partitions them into batches

For v2.11.0, the orchestrator is lightweight (~2048 max_tokens, fast model). It doesn't generate content — it plans.

### Specialist Agents

Each specialist has a focused system prompt (~500-1000 tokens) and produces typed IR nodes:

#### @sections-writer
- **Input:** project_context, tech_stack, autonomy_level
- **Output:** `Section[]` (CLAUDE.md sections following the mandatory template)
- **System prompt:** CLAUDE.md template rules, section ordering, line budget per section
- **Max tokens:** 4096

#### @command-writer  
- **Input:** project_context, command list from plan, Section[] (from Phase A), autonomy_level
- **Output:** `CommandNode[]`
- **System prompt:** `!` shell integration rules, `$ARGUMENTS` syntax, orchestration patterns
- **Max tokens:** 4096 (or batched if >10 commands)

#### @agent-writer
- **Input:** project_context, agent list from plan, RuleNode[] (from Phase A), autonomy_level
- **Output:** `AgentNode[]` (with `modelRouting` YAML frontmatter)
- **System prompt:** YAML frontmatter conventions, persona design, `disallowedTools`, model tiering
- **Max tokens:** 4096 (or batched if >8 agents)

#### @rule-writer
- **Input:** project_context, tech_stack
- **Output:** `RuleNode[]` (with path-scoped YAML frontmatter where applicable)
- **System prompt:** path scoping syntax, security baseline, constraint format
- **Max tokens:** 2048

#### @doc-writer
- **Input:** project_context
- **Output:** `DocNode[]` (DECISIONS.md, LEARNINGS.md, SPRINT.md templates)
- **System prompt:** template format, acceptance criteria structure
- **Max tokens:** 2048

#### @skill-writer
- **Input:** project_context, skill list from plan
- **Output:** `SkillNode[]` (TDD skill, etc.)
- **System prompt:** SKILL.md format, 3-phase patterns
- **Max tokens:** 2048

#### @linker (Phase C)
- **Input:** all generated nodes from Phase A + B
- **Output:** patched nodes with fixed cross-references
- **System prompt:** validate `@agent` mentions in commands, command references in agents, rule scope coverage
- **Max tokens:** 2048

### Batch Execution Engine

Reuse the `runWithConcurrency` pattern from evolve:

```typescript
async function executePlan(
  plan: CompilationPlan,
  config: KairnConfig,
  concurrency: number,
  onProgress?: (progress: CompileProgress) => void,
): Promise<HarnessIR> {
  const ir = createEmptyIR();
  const completed = new Map<string, boolean>();

  for (const phase of plan.phases) {
    // Wait for dependencies
    for (const dep of phase.dependsOn) {
      if (!completed.get(dep)) {
        throw new Error(`Phase ${phase.id} depends on incomplete phase ${dep}`);
      }
    }

    // Execute all agents in this phase with concurrency limit
    const results = await runWithConcurrency(
      phase.agents.map(task => () => executeAgent(task, config, ir)),
      concurrency,
    );

    // Merge results into IR
    for (const result of results) {
      mergeIntoIR(ir, result);
    }

    completed.set(phase.id, true);
  }

  return ir;
}
```

**Concurrency control:**
- API key users: concurrency 3-4
- OAuth (Claude Code subscription) users: concurrency 1-2 (stricter rate limits)
- Configurable via `kairn init` or `--concurrency` flag

### Truncation Detection

In `callLLM()`, check `response.stop_reason`:

```typescript
// Anthropic
if (response.stop_reason === 'max_tokens') {
  throw new TruncationError(
    `Response truncated at ${maxTokens} tokens. Agent: ${agentName}`,
    { agentName, tokensUsed: maxTokens }
  );
}
```

Each specialist handles truncation independently:
- Re-call with higher `maxTokens` (up to 2x)
- Or split the item list into smaller batches
- Or fall back to concise mode for that specific agent

### HarnessIR as Compilation Output

The compilation pipeline produces `HarnessIR` directly. The existing renderer (`src/ir/renderer.ts`) emits files. The existing adapters consume the rendered files.

**Migration path:**
1. `compile()` returns `HarnessIR` instead of `EnvironmentSpec`
2. `EnvironmentSpec` retains metadata (`id`, `intent`, `created_at`, `tools`) but references `HarnessIR` for content
3. Adapters (`claude-code.ts`, `hermes-agent.ts`) consume `HarnessIR` via renderer
4. `evolve` loop no longer needs to re-parse compiled environments — it receives `HarnessIR` directly

```typescript
// New EnvironmentSpec
interface EnvironmentSpec {
  id: string;
  intent: string;
  created_at: string;
  name: string;
  description: string;
  autonomy_level: 1 | 2 | 3 | 4;
  tools: ToolSelection[];
  harness: HarnessIR;  // ← was: flat string fields
}
```

### Scale Path: Swarm Compilation

For Isara-scale deployments (100-1000+ agents), the same architecture scales:

```
@orchestrator: "You need 200 forecaster agents with 5 specializations"
→ CompilationPlan: [
    { agent: 'agent-writer', items: ['equity-analyst-1', ..., 'equity-analyst-40'] },
    { agent: 'agent-writer', items: ['macro-analyst-1', ..., 'macro-analyst-40'] },
    { agent: 'agent-writer', items: ['sentiment-analyst-1', ..., 'sentiment-analyst-40'] },
    ...
  ]
→ Batch execution with concurrency limit
→ Same code path, just more batches
```

The batch executor doesn't care if there are 11 or 200 agents. Token budget per batch stays constant.

## Cost Analysis

### Before (monolithic)
- Pass 1: ~2K input + 1K output tokens
- Pass 2: ~5K input + 8-16K output tokens (often truncated)
- **Total: ~7K input + 9-17K output tokens**

### After (multi-agent)
- Pass 1: ~2K input + 1K output (unchanged)
- Pass 2 orchestrator: ~2K input + 0.5K output
- Phase A (3 agents): ~3 × (1.5K input + 1.5K output) = ~9K total
- Phase B (2 agents): ~2 × (2K input + 2K output) = ~8K total  
- Phase C linker: ~3K input + 0.5K output
- **Total: ~20K input + 10K output tokens**

Input tokens increase ~2.5x (system prompts repeated per agent), but output tokens are distributed safely below per-call limits. With Anthropic prompt caching (already implemented for evolve), repeated system prompts cost ~10% of base rate.

**Net cost: ~30-50% more tokens, but zero truncation failures and higher output quality.**

Wall-clock time: **faster** (parallel execution) despite more total tokens.

## Testing Strategy

### Unit tests
- `plan.ts`: orchestrator produces valid CompilationPlan from various skeleton types
- `agents/*.ts`: each specialist produces correctly typed IR nodes
- `batch.ts`: concurrency control, dependency ordering, error propagation
- `linker.ts`: cross-reference detection and patching
- `compile.ts`: end-to-end compilation produces valid HarnessIR

### Integration tests
- Round-trip: compile → render → parse → compare IR
- Adapter compatibility: compile → claude-code adapter → files → parse → same IR
- Evolve compatibility: compile → evolve mutations → render → valid harness
- Error scenarios: truncation recovery, agent failure isolation, rate limiting

### Regression tests
- All existing `compile.ts` tests adapted for new pipeline
- Ensure identical output quality (human review of generated harnesses)

## Migration

### EnvironmentSpec backward compatibility

Saved environments (in `~/.kairn/envs/`) use the old `EnvironmentSpec` shape. Migration:
1. Old `EnvironmentSpec` with string `harness.claude_md` → parse into `HarnessIR` on load
2. New `EnvironmentSpec` with `HarnessIR` → use directly
3. Version field in EnvironmentSpec to distinguish

### Evolve workspace compatibility

Existing `.kairn-evolve/` workspaces use file-based harnesses. No change needed — the IR parser/renderer bridge already exists from v2.7.0.

---

## Out of Scope for v2.11.0

- **Swarm compilation (100+ agents):** Architecture supports it, but UX and batch scheduling are v3.x
- **Per-command/per-agent granularity:** Each specialist batch-generates its category. Per-item calls are a future optimization.
- **Experience replay across compilations:** Cross-project learning is a v3.x feature
- **Custom specialist agents:** User-defined compilation agents are a v3.x feature

---

## Success Criteria

1. `kairn describe` never produces truncated JSON (the original bug is gone)
2. Generated harness quality ≥ current monolithic output (human-reviewed on 5 test projects)
3. Compilation wall-clock time ≤ current (parallelism offsets extra calls)
4. `compile()` returns `HarnessIR`, not flat strings
5. All existing tests pass with adapted assertions
6. `kairn evolve` works on environments compiled with the new pipeline
7. Both adapters (claude-code, hermes-agent) produce identical file output from `HarnessIR`
